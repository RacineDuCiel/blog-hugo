#certifications/CASP

Core Objective 3.2: Given a scenario, configure and implement endpoint security controls. This chapter focuses on Defense in Depth for endpoints, applying layers of security through hardening, trusted OS concepts, and compensating controls. 1. Hardening Techniques (Reducing Attack Surface)

  • Goal: Make endpoints harder to compromise by removing unneeded services/features and securing configurations. Requires standard methodology, OS-specific baselines.
  • Golden Image / System Cloning: Creating a master, hardened image for consistent deployment. Establishes a baseline.
  • Physical Security: Foundational. Secure systems physically. Train users (power off, password screensavers, auto-logoff).
  • Lifecycle Management:
    • EOL (End-of-Life): Remove hardware reaching end of usability from the network.
    • EOS (End-of-Support): Replace software no longer supported by the manufacturer.
  • Boot Loader Protections: Secure systems before the OS fully loads.
    • Secure Boot: UEFI feature using manufacturer-trusted software/keys (X.509 certs) to ensure only signed bootloaders run. Blocks pre-boot malware.
    • Measured Launch/Boot: Works with TPM & Secure Boot. Firmware logs the boot process; logs can be sent to a trusted server for health attestation/verification before allowing OS components to execute.
    • IMA (Integrity Measurement Architecture): Verifies integrity/trust of Linux OS components during boot.
    • BIOS/UEFI Protection: UEFI replaces/augments BIOS, acts like a pre-OS, designed to block rootkits. Requires Secure Boot enablement.
  • General Host Hardening:
    • Application Approved List (Whitelist): Only explicitly listed applications/entities can run/access. Implicit “deny all.”
    • Application Block/Deny List (Blacklist): Explicitly listed applications/entities cannot run/access. Implicit “allow all.”
    • Security/Group Policy (Windows): Centralized configuration management via Active Directory. Enforces consistent settings (Password Policy, Lockout Policy, Kerberos, Audit, User Rights, Security Options).
    • Attestation Services: Validating something (like boot process or hardware state) as true. Can be hardware (TPM), software, or hybrid. TPM enables tamper-resistant full-disk encryption.
    • NX (No-eXecute) / XN (eXecute Never) Bit: CPU feature segregating memory areas for code and data, preventing code execution from data segments. Protects against certain exploits.
      • Intel: XD (eXecute Disable)
      • AMD: EVP (Enhanced Virus Protection)
      • ARM: XN (eXecute Never)
    • Secure Enclaves / Memory Encryption: Hardware-level secure application execution. Data is encrypted in main memory, decrypted only within the secure hardware enclave. Protects even if OS kernel is compromised.
    • ASLR (Address Space Layout Randomization): Security technique randomizing memory locations of key process areas (executables, libraries, stack, heap). Makes buffer overflow exploitation much harder (attacker needs exact address). Requires application compilation support (now common). Note: Can have configuration issues/compatibility challenges.
    • Patching & Updates: Regularly apply OS and application patches to fix known vulnerabilities. Use standard images (golden images) and vulnerability assessment tools (Nessus, SAINT, Retina) to identify needed patches. Test before deploying.
    • Removing Unnecessary Services/Applications: Core hardening principle. Disable/remove unneeded software, services, ports. Reduces potential entry points. (e.g., Bastille Linux tool for automating Linux hardening).
    • Command Shell Restrictions: Limit or disable user access to command prompts (cmd, bash) unless necessary. Prevents misuse or privilege escalation via commands. Note: Can sometimes be bypassed (e.g., StickyKeys exploit example).
    • Warning Banners: Display messages upon login/access informing users of acceptable use policies, monitoring, lack of privacy expectations, and consequences of violations. Needs legal/HR approval.
    • Restricted Interfaces: User profiles limiting available programs, menus, functions to only what’s needed for a specific role/task (e.g., kiosk).
    • Dedicated Interfaces: Physical network ports dedicated to specific types of traffic (e.g., separate NIC/VLAN for wireless traffic).
    • Out-of-Band (OOB) Management: Using a separate, dedicated channel (often a dedicated NIC/network) for managing devices (servers, network gear), isolated from production data traffic.
    • Management Interface: Hardware-level interface (often using OOB NIC) allowing remote management even if the OS is down/unresponsive.
    • Data Interface: Used with databases to generate reusable process templates for manipulating similar data types across clients.
    • Scripting & Replication: Using scripts (e.g., PowerShell) to automate tasks like configuration, disaster recovery setup (replicating objects), and patch management.
    • Standard Operating Environment (SOE): Using a consistent, standard OS build across the organization. Reduces TCO, simplifies testing, patching, and security management.

⠀2. Hardware Security Technologies

  • HSM (Hardware Security Module): Secure cryptoprocessor device managing cryptographic keys (generation, signing, validation). High-assurance key management.
  • TPM (Trusted Platform Module): Specialized chip on motherboard providing hardware-based security functions:
    • Hardware Authentication: Authenticates the computer itself (not the user).
    • Secure Key Storage: Creates and stores crypto keys securely.
    • Platform Integrity Measurement: Used by Measured Boot to securely report on the boot environment.
    • Tamper Resistance: Enables features like full-disk encryption tied to the specific hardware (drive won’t work easily in another machine).
  • SED (Self-Encrypting Drive): HDD or SSD that automatically encrypts/decrypts data written to/read from it at the hardware level. Encryption is always on and transparent to the OS/user.

⠀3. Trusted Operating Systems (Trusted OS)

  • Definition: An OS designed and tested to meet specific security standards (like Common Criteria) enabling it to handle information at different security levels (multilevel security).
  • TCB (Trusted Computer Base): The totality of protection mechanisms within a computer (hardware, firmware, software, controls, processes) responsible for enforcing the security policy. The core trusted part of the system.
  • Security Kernel: The core part of the TCB that handles all access requests between subjects (users/processes) and objects (files/resources). Designed to be small, verifiable, and always invoked.
  • Core Attributes of Trusted OS:
    • Hardware Protection
    • Long-Term Protected Storage (across reboots)
    • Isolation (between processes)
    • Separation of User vs. Supervisor (Kernel) modes.
  • Validation Standards:
    • TCSEC (Orange Book): Deprecated. US DoD standard, focused on Confidentiality. Levels D (Minimal) to A (Verified).
    • ITSEC: Deprecated. European standard, evaluated Confidentiality, Integrity, Availability (CIA). Functionality (F) & Assurance (E) ratings.
    • Common Criteria (CC) (ISO 15408): Current Global Standard. Evaluates security functions and provides EAL (Evaluation Assurance Level) ratings (EAL1 to EAL7). Higher EAL = higher assurance. Can be “augmented” (e.g., EAL4+).
  • MAC (Mandatory Access Control): Access decisions are enforced by the OS based on security labels/levels (e.g., Classified, Secret, Top Secret) assigned to subjects and objects. Users cannot override (unlike DAC).
  • Trusted OS Examples:
    • SELinux (Security-Enhanced Linux): Adds MAC capabilities to the Linux kernel. NSA/Red Hat collaboration.
    • SEAndroid (Security-Enhanced Android): Brings SELinux MAC benefits to the Android kernel, enhancing application sandboxing. Fully enforced since Android 5.0.
    • Trusted Solaris: Provides MAC and features like “immutable zones” (read-only file system protection).
  • Kernel vs. Middleware:
    • Kernel: The core of the OS, providing fundamental services. SELinux/SEAndroid operate here.
    • Middleware: Software layer providing services to applications beyond the OS kernel (e.g., Android libraries for data storage, display, web browsing). “Software glue.”

⠀4. Compensating Controls

  • Definition (per CompTIA Objectives): Controls implemented to satisfy a security requirement when the primary/desired control is not feasible or available. (Note: Often overlaps with Mitigating Controls which aim to reduce likelihood/impact). Used to enhance endpoint security layers.
  • Examples:
    • Antivirus (AV): Detects/removes known malware. Essential endpoint protection.
    • Application Controls: Security measures within applications (e.g., input validation, authentication checks, authorization).
    • HIDS/HIPS (Host-based Intrusion Detection/Prevention System): Monitors endpoint activity for malicious behavior; HIPS can block detected threats. Provides visibility for SOC.
    • Host-Based Firewall: Software firewall on the endpoint controlling network traffic in/out of that specific host. Adds layer beyond network firewall.
    • EDR (Endpoint Detection and Response): Advanced endpoint security solution providing real-time monitoring, threat detection (especially APTs), investigation tools, and automated response capabilities. Examples: Microsoft Defender for Endpoint, Carbon Black, Palo Alto Traps.
    • Redundant Hardware: Using duplicate components (power supplies, NICs, disks) to improve availability (HA) and reduce downtime risk.
    • Self-Healing Hardware/Software: Systems designed to automatically detect, respond to, and potentially fix failures or security issues (e.g., hardware automatically recovering, software rolling back a bad patch).
    • SED (Self-Encrypting Drives): (Also listed under Hardware) Automatic hardware-level disk encryption.
    • UEBA (User and Entity Behavior Analytics): Analyzes user and system/entity behavior patterns to detect anomalies indicating threats (insider threats, compromised accounts/devices).

⠀Exam Essentials Summary:

  • Endpoints face different risks; apply hardening/controls accordingly.
  • Apply multiple hardening techniques (not one-size-fits-all) based on effectiveness for the specific scenario/risk. Layer controls (Defense in Depth).
  • Understand the purpose of each compensating control and how it mitigates specific endpoint risks (e.g., EDR for APTs, HIPS for intrusions, Host Firewall for network threats, Redundancy for availability).
  • Know the core concepts of Trusted OS (TCB, Kernel, MAC) and their advantages (smaller attack surface, stronger enforcement).
  • Recognize hardware security features (TPM, HSM, SED, NX Bit) and their roles.
  • Be aware of lifecycle management (EOL/EOS) importance.
