Core Objectives Covered: 2.3 (Vuln Mgmt Activities), 2.4 (Vuln Assess & Pen Test Tools/Methods), 2.5 (Analyze Vulns & Mitigate Risk), 2.6 (Processes to Reduce Risk) Foundational Terminology (Recap)

  • Asset: Item of value (data, hardware, software, property, sometimes employees).
  • Risk: Probability/likelihood of a threat exploiting a vulnerability.
  • Vulnerability: Weakness in hardware, software, design, or procedure exploitable by a threat.
  • Threat: Agent, condition, or circumstance potentially causing harm, loss, or compromise.
  • Motivation: Driving force behind threat actor (money, fame, challenge, etc.).
  • Risk Source: Origin of risk (Internal: employee, failure; External: disaster, human-made event).

⠀Objective 2.3: Perform Vulnerability Management Activities 1. Vulnerability Management Lifecycle (Cyclic Process - Fig 4.1) 1 Discovery: Identify assets on the network (inventory is crucial - CIS Control #1). Use scanning tools (ping sweeps, port scans). Cannot protect what you don’t know. 2 Prioritization: Determine asset criticality (e.g., Domain Controller vs. marketing file server). Focus resources on high-value/high-risk assets. 3 Assessment: Scan prioritized assets for vulnerabilities (OS, open ports, services, application versions). 4 Reporting: Generate reports for different audiences (Management: trends, remediation plans, budget needs; Asset Owners: technical details, fixes). 5 Remediation: Fix the identified vulnerabilities (patching, configuration changes). 6 Verification: Rescan to confirm vulnerabilities are resolved.

  • Vulnerability Window/Time: Time between vulnerability discovery and patching. Shorter is better.
  • VMaaS (Vulnerability Management as a Service): Outsourcing vulnerability management tasks. Benefits: actionable intelligence, visibility, proactive approach, compliance aid (e.g., SOX), frees internal resources.

⠀2. Vulnerability Scanning Techniques

  • Credentialed vs. Non-credentialed (Authorized vs. Unauthorized):
    • Non-credentialed: Scans from outside perspective (like an attacker). Uses basic probes (e.g., ICMP ping). Shows external exposure. Faster.
    • Credentialed: Uses service account credentials to log into assets. Provides deep insight (patch levels, installed software/versions, configuration). More resource-intensive, takes longer. Risk: Exposes credentials.
  • Agent-based vs. Server-based:
    • Server-based: Central scanning server initiates scans (often scheduled). Standard approach.
    • Agent-based: Small software agent installed on endpoints. Reports back at intervals. Eliminates need for credentialed scans, reduces credential exposure risk, better for remote/off-network assets.
  • Active vs. Passive Scanning (OS Fingerprinting):
    • Active: Sends packets to target, analyzes response (e.g., Nmap -O). More accurate but detectable by IDS/IPS/Firewalls. Can potentially crash targets.
    • Passive: Analyzes captured network traffic (.pcap files) without sending packets to the target (e.g., Ettercap). Less accurate, avoids detection, avoids interference with hosts. Mitigated by disabling promiscuous mode. Wireshark is used to create .pcap files.

⠀3. Criticality Ranking & Vulnerability Databases

  • CVE (Common Vulnerabilities and Exposures): Standardized list/database of publicly known vulnerabilities. Each has a unique CVE ID, description, references. Maintained by MITRE, searchable via NVD.
  • CVSS (Common Vulnerability Scoring System): Standardized scoring system (0.0 - 10.0) indicating vulnerability severity/criticality. Considers scope, access vector, complexity, authentication, CIA impact. Used in NVD. (See Figure 4.2 for qualitative scale: None, Low, Medium, High, Critical).
  • NVD (National Vulnerability Database): U.S. government repository (maintained by NIST) of vulnerability data, including CVEs, CVSS scores, CPEs, etc. Is the U.S. gov content repository for SCAP.

⠀4. Security Content Automation Protocol (SCAP)

  • Purpose: Standardized methodology for vulnerability management, measurement, policy compliance automation. Uses multiple components together.
  • SCAP Products: Content Producers (generate SCAP data) vs. Content Consumers (process SCAP data).
  • SCAP Scanner: Tool applying security standards/baselines (often from NVD) to scan systems and compare configurations/patches against the baseline.
  • SCAP Components:
    • OVAL (Open Vulnerability and Assessment Language): XML language to express checks for machine state (config issues, patches, versions). Standardizes assessment definitions. OVAL Interpreter is a tool to run OVAL definitions. Community-driven (CIS maintains).
    • XCCDF (Extensible Configuration Checklist Description Format): XML language for security checklists, benchmarks, configuration specifications. Standardizes checklist format.
    • CPE (Common Platform Enumeration): Standardized naming convention/dictionary for IT platforms (hardware, OS, applications). Allows tools to identify software/hardware classes.
    • CVE (Common Vulnerabilities and Exposures): (See above) - Used within SCAP framework.
    • CVSS (Common Vulnerability Scoring System): (See above) - Used within SCAP framework.
    • CCE (Common Configuration Enumeration): Standardized identifiers for system configuration issues (links guidance docs to machine checks). Used in NIST guides, CIS Benchmarks, DISA STIGs. (See Figure 4.3 example).
    • ARF (Asset Reporting Format): Standardized XML model for expressing information about assets and their relationships. Vendor-neutral reporting.

⠀5. Self-Assessment vs. Third-Party Vendor Assessment

  • Self-Assessment: Internal team reviews policies, procedures, controls. Suitable for low-risk orgs. Requires cross-functional team, management support, funding. Data sources: asset owners, IT/Sec admins, HR, legal, audits, records, advisories, ISACs. Considers internal/external threats (natural, malicious code, physical breach, hackers) and insider/outsider risks (resource abuse, theft, alteration, disclosure, embezzlement, destruction, unauthorized entry).
  • Third-Party Assessment: Conducted by neutral, external organization. Provides independent perspective. Often needed for higher risk, compliance (HIPAA, PCI-DSS), or assessing vendor/supplier risk (supply chain security). Usually based on frameworks (NIST, ISO). Checks vendor data handling, encryption, access controls.

⠀6. Patch Management

  • Purpose: Apply updates (patches) to fix vulnerabilities or add functionality. Vital for security.
  • Patch Management Lifecycle (Fig 4.4): 1 Audit/Discover: Scan environment to identify needed patches. 2 Test: Apply patches in a non-production environment first to check for adverse effects (regression issues). Crucial step. 3 Deploy: Roll out tested patches to production systems (prioritize critical/vulnerable systems). Use tools like WSUS or third-party solutions. 4 Verify: Confirm patches applied successfully and systems are stable.
  • Importance: Highlighted by major exploits like EternalBlue (MS17-010) used by WannaCry, NotPetya, WannaMine (cryptojacking). Failure to patch leads to preventable breaches.
  • Scope: Includes OS (Windows, macOS, Linux) and third-party applications (Adobe, Java, browsers).
  • Challenge: Balancing patching speed (reduce vulnerability window) vs. change management/testing (avoid breaking systems). Documented strategy is key.

⠀7. Information Sources for Vulnerability/Threat Intelligence

  • Advisories: Shared by government (e.g., CISA) or vendors. Inform about specific threats/risks. (e.g., CIS Advisories).
  • Bulletins: Periodic summaries of vulnerabilities, risks, threats, patches. (e.g., US-CERT/CISA Bulletins).
  • Vendor Websites: Primary source for product-specific patches and security updates.
  • ISACs (Information Sharing and Analysis Centers): Industry-specific groups for sharing cyber threat info and best practices (e.g., IT-ISAC, Financial Services ISAC, Health ISAC).
  • News Reports/Security Websites: Stay current on attacks, trends, research. (Examples: The Hacker News, Threatpost, Dark Reading, Wired, CNET, CERT/CC, SecuriTeam, SecurityTracker, SecurityFocus, Exploit-DB).
  • Conferences/Blogs: Learning from experts and community.

⠀Objective 2.4: Use Appropriate Vuln Assessment & Pen Test Methods/Tools 1. Assessment & Testing Methods

  • Static Analysis (SAST - Static Application Security Testing): Analyzing source code without executing it. Fast feedback, pinpoints location. Finds issues like buffer overflows, race conditions, vulnerable functions (e.g., C functions: scanf, strcpy, gets). Tools: RATS, Flawfinder. Can have false positives.
  • Dynamic Analysis (DAST - Dynamic Application Security Testing): Testing software while it’s running. Observes behavior, execution logic, live data. Finds runtime issues like SQL injection, XSS.
    • SAST + DAST recommended for comprehensive testing.
  • Fuzz Testing (Fuzzing): Inputting invalid, unexpected, or random data (“fuzz”) into an application to cause crashes or unexpected behavior, revealing bugs/vulnerabilities. Simple test design. Tools: SPIKE, WFuzzer, Mangle, ISIC.
  • Reverse Engineering: Deconstructing software/hardware to understand its inner workings. Can reveal weaknesses beyond standard testing.
  • Side-Channel Analysis: Exploiting information leaked through physical implementation characteristics (timing, power consumption, EM radiation) rather than crypto flaws. Can reveal keys or confidential data. (Contrast with Covert Channel: Intentionally hidden communication path).
  • Wireless Vulnerability Scan/Assessment (Site Survey): Examining wireless networks (Wi-Fi, Bluetooth, Zigbee, RFID). Checks encryption (WEP - broken, WPA/TKIP - deprecated, WPA2/AES - common, WPA3/SAE - current best), DoS potential, ARP poisoning, on-path attacks. Tools: Kismet, NetStumbler, Aircrack-ng (WEP/WPA/WPA2/WPA3 cracking), Wireshark, commercial tools (SolarWinds NPM, ManageEngine NetFlow Analyzer).
  • Software Composition Analysis (SCA): Identifying and analyzing risks associated with open-source and third-party components/libraries used within software. Checks for age, known vulnerabilities, license issues. (OWASP Top 10 item).
  • Pivoting: Using a compromised system as a launch point (“pivot”) to access other systems/networks not directly reachable by the attacker. Tools: Metasploit, Netcat.
  • Post-Exploitation: Actions taken after gaining initial access. Includes:
    • Privilege Escalation: Gaining higher-level access (user to admin/root). Vertical (low to high user) vs. Horizontal (user to another user). Exploits OS/app flaws. (Rings of protection concept).
    • Persistence: Maintaining access over time (creating accounts, planting backdoors, tunnels).
  • Footprinting (Reconnaissance): Gathering information about a target.
    • Passive: No direct interaction (website review, job postings, WHOIS, DNS records, financial filings - SEC/D&B, social media, metadata - FOCA tool, Wayback Machine). Tools: Google Dorking, whois.domaintools.com, dnsdumpster.com.
    • Active: Direct interaction (calling company, scanning).

⠀2. Assessment & Testing Tools

  • SCAP Scanner: Applies SCAP standards/content for automated checks (already covered).
  • Network Traffic Analyzer / Protocol Analyzer / Packet Sniffer: Captures and analyzes network packets. Wireshark (most common, GUI, Windows/Linux), tcpdump (command-line Linux), WinDump (command-line Windows). Needs promiscuous mode NIC. Uses SPAN/mirror ports on switches.
  • Vulnerability Scanner: Automated tools to probe for weaknesses. Nessus (popular commercial), OpenVAS (open-source fork of Nessus), SAINT, GFI LanGuard, Retina. Categories: System, Application, Source Code scanners.
  • Port Scanner: Probes TCP/UDP ports to determine status (open, closed, filtered). Nmap (most popular, CLI/GUI). Scan Types: TCP Connect (-sT), TCP SYN (-sS, half-open), UDP (-sU), FIN, NULL, ACK, XMAS. OS Fingerprinting (-O). Banner Grabbing (telnet to port, Nmap, Netcat).
  • HTTP Interceptor (Web Proxy): Intercepts, inspects, modifies HTTP/HTTPS traffic between client and server. Used for web app testing. Burp Suite (industry standard, free/paid), OWASP ZAP (free), WebScarab, Paros Proxy.
  • Exploit Framework: Platform for developing, testing, using exploit code. Metasploit Framework (most popular, free/paid).
  • Password Cracker: Attempts to recover passwords from hashes/transmissions. John the Ripper (multi-platform), L0phtCrack (Windows), Cain & Abel (Windows, multiple functions), Ophcrack (rainbow tables). Methods: Dictionary, Hybrid, Brute-Force. Defenses: Salting, strong password policies. GPU cracking increases speed.
  • Other Mentioned Tools: Netcat (network utility), Hping (packet crafting), Kismet/Aircrack-ng (wireless), Snort (IDS), Kali Linux (pentest distro), PuTTY (SSH client), Microsoft SDL Threat Modeling Tool, Forensic tools (Helix, EnCase, FTK), Data Visualization tools (Tableau, Qlikview).
  • Tool Integrity: Verify tool downloads using hash values (MD5, SHA1, SHA256) provided by reputable sources.

⠀3. Assessment Requirements & Considerations

  • Scope of Work / Rules of Engagement (RoE): Formal agreement defining targets (IP ranges, apps), methods allowed/disallowed, timing, points of contact, reporting requirements. CRITICAL: Obtain written permission BEFORE starting. Keep authorization letter handy.
  • Invasive vs. Non-invasive: Determine acceptable level of intrusiveness (e.g., allow DoS testing?).
  • Asset Inventory: Need accurate list of assets within scope.
  • Permissions and Access: Obtain necessary credentials/authorization for credentialed scans or internal tests.
  • Corporate Policy Considerations: Adhere to company security/testing policies.
  • Facility Considerations: Physical location, access requirements for onsite testing.
  • Physical Security Considerations: Guards, locks, cameras affecting onsite tests.
  • Rescan for Corrections/Changes: Verify remediation efforts.

⠀4. Penetration Testing Methodology (General Steps) 1 Planning & Scoping: Define goals, RoE, get written permission. 2 Footprinting/Reconnaissance: Gather information (passive/active). 3 Scanning: Identify live hosts, open ports, services, vulnerabilities (port scanners, vuln scanners). 4 Gaining Access (Exploitation): Exploit identified vulnerabilities (Metasploit, social engineering, password attacks). 5 Post Exploitation: Escalate privileges, pivot to other systems. 6 Persistence: Maintain access if required by scope (backdoors, accounts). 7 Covering Tracks: Remove tools, logs, accounts (if required by RoE). 8 Analysis & Reporting: Analyze findings, determine recommendations, write detailed report (vulnerabilities, risks, recommended controls). Present findings to client.

  • Keep Client Informed: Communicate findings throughout the test.
  • Pen Test Types: No Knowledge (Black Box), Full Knowledge (White Box), Partial Knowledge (Gray Box).
  • OSSTMM (Open Source Security Testing Methodology Manual): Framework for security testing.

⠀5. Other Assessment Types

  • Security Audit: Independent review of controls, policies, procedures against standards/requirements (internal or external).
  • Security Code Review: Manual or automated examination of source code for flaws.
  • Tabletop Exercise: Discussion-based walkthrough of incident scenarios to test plans/responses.
  • Vulnerability Assessment vs. Penetration Test:
    • Vuln Assessment: Breadth-focused. Identifies potential weaknesses using automated tools. Less intrusive. Often done regularly. Goal: Find vulnerabilities.
    • Pen Test: Depth-focused. Attempts to exploit vulnerabilities to simulate an attack. More intrusive. Goal: Determine if vulnerabilities are exploitable and assess impact.

⠀Objective 2.5: Analyze Vulnerabilities and Recommend Risk Mitigations (Common Vulnerabilities & Attacks)

  • Memory Safety Issues:
    • Buffer Overflow: Writing more data than buffer holds, overwriting adjacent memory (e.g., return address). Can lead to crash or code execution. Defense: Bounds checking, safer functions, OS protections (ASLR, DEP/NX). Vulnerable C functions: strcpy, gets, scanf, sprintf.
    • Integer Overflow: Arithmetic operation results in a value exceeding variable’s storage capacity, wrapping around (often to 0 or negative). Can bypass security checks or cause crashes. Defense: Use larger data types, input validation.
    • Memory Leaks: Application fails to release allocated memory it no longer needs. Leads to performance degradation and eventual crash (resource exhaustion). Common in C/C++. Defense: Proper memory management (e.g., free calls), garbage collection (in some languages).
    • Data Remnants: Residual data left on media after deletion/formatting. Risk of sensitive info recovery. Defense: Secure wiping/sanitization, physical destruction.
  • Concurrency Issues:
    • Race Condition (Time-of-Check-to-Time-of-Use - TOC/TOU): Exploiting the delay between checking a resource’s state (e.g., file permissions) and using it. Attacker modifies resource in between check and use. Hard to detect/exploit. Defense: Atomic operations, locking mechanisms.
  • Input Validation Issues:
    • Injection Attacks (OWASP #3): Inserting malicious data into input fields, interpreted as commands by backend systems.
      • SQL Injection (SQLi): Target databases.
      • Command Injection: Execute OS commands.
      • LDAP Injection: Target LDAP directories.
      • XML Injection (XXE): Target XML parsers.
    • Cross-Site Scripting (XSS): Injecting malicious scripts into web pages viewed by other users. Steals sessions, redirects users. Defense: Input sanitization, output encoding, Content Security Policy (CSP).
    • Directory Traversal: Accessing files/directories outside the intended web root using ../. Defense: Input validation, proper path handling.
  • Authentication & Authorization Issues:
    • Broken Authentication (OWASP #7): Flaws in login mechanisms, session management, password handling. Leads to account takeover. Defense: Strong passwords, MFA, secure session management, protection against credential stuffing/brute-force.
    • Broken Access Control (OWASP #1): Users can access resources/perform actions they shouldn’t be authorized for (e.g., insecure direct object references, missing function-level checks). Defense: Enforce least privilege, deny by default, proper authorization checks server-side.
    • Authentication Bypass: Finding ways to circumvent login entirely.
    • Cross-Site Request Forgery (CSRF/XSRF): Forcing an authenticated user’s browser to send unintended requests to a web application. Defense: Anti-CSRF tokens, SameSite cookies.
  • Cryptographic Failures (OWASP #2):
    • Weak Ciphers/Implementations: Using outdated/flawed algorithms (DES, MD5, SHA1) or weak modes of operation.
    • Insufficient Key Length: Keys too short for brute-forcing.
    • Improper Key Management: Storing keys insecurely.
    • Certificate Errors: Expired, self-signed, mismatched names, weak signature algorithms, untrusted CAs. CRL/OCSP issues. Wildcard certificate risks (compromise of one server exposes key for all). Defense: Use current strong algorithms (AES, SHA-256+), adequate key lengths, secure key storage (HSMs), proper certificate validation. Kerckhoffs’ Principle.
  • Configuration & Design Issues:
    • Security Misconfiguration (OWASP #5): Missing hardening, default credentials, unnecessary features/ports enabled, verbose errors, improper permissions.
    • Insecure Design (OWASP #4): Lack of threat modeling, secure design patterns not used from the start. Security added later (“bolted on”) is less effective. Security by Design/Default/Deployment concepts.
    • Vulnerable and Outdated Components (OWASP #6): Using libraries, frameworks, OS with known vulnerabilities (often via SCA). Includes EOL/EOS software. Defense: Patch management, SCA tools, inventory management.
    • Software and Data Integrity Failures (OWASP #8): Lack of protection against modification (e.g., insecure update processes, unverified serialized data). Defense: Digital signatures, integrity checks.
    • Security Logging and Monitoring Failures (OWASP #9): Insufficient logging, ineffective alerting, lack of incident response capability.
    • Server-Side Request Forgery (SSRF) (OWASP #10): Attacker forces server to make requests to unintended internal/external resources.
    • Improper Headers: Missing or insecure HTTP security headers (e.g., HSTS, X-Frame-Options, CSP).
    • Information Disclosure: Leaking sensitive info via error messages, comments, metadata (EXIF in photos, FOCA tool).
    • Poor Exception Handling: Revealing sensitive system details in error messages.
    • Use of Unsafe Functions: Using functions known to be vulnerable (e.g., C functions susceptible to buffer overflows).
  • Client-Side Issues:
    • Client-Side Processing: Relying on client (browser) for security checks (validation, logic) which can be bypassed. Technologies: JavaScript, Java Applets (bytecode), AJAX, HTML5.
    • Browser Extensions: Can introduce vulnerabilities or be malicious. Flash/ActiveX (Deprecated, highly insecure).
  • Network & Protocol Issues:
    • Denial-of-Service (DoS)/DDoS: Overwhelming resources (bandwidth, CPU, memory). Attacks: Smurf, SYN Flood, Ping of Death, Teardrop, Land, Trinoo, LOIC. Defense: Firewalls, IPS, rate limiting, flood guards, DDoS mitigation services.
    • Interception Attacks (On-Path): Intercepting/modifying traffic. Defense: Encryption (TLS/SSL, SSH, VPNs).
    • VLAN Hopping: Gaining access to traffic on other VLANs. Defense: Proper switch configuration (disable DTP, native VLAN security).
    • BGP/Route Hijacking: Maliciously altering internet routing tables. Defense: RPKI, BGP monitoring.
  • Virtualization Issues:
    • VM Escape: Breaking out of a VM to access the host OS or other VMs.
    • VM Hopping: Moving between different VMs on the same host. Defense: Patch hypervisor, secure VM configurations, network segmentation.
  • Other:
    • Social Engineering: Manipulating people (phishing, vishing, pretexting). Exploits trust, uses OSINT. Defense: User training, security awareness, verification procedures.
    • Code Reuse / Third-Party Libraries: Can inherit vulnerabilities if not properly vetted/managed (SCA importance).

⠀Objective 2.6: Use Processes to Reduce Risk 1. Proactive and Detection Processes

  • Threat Hunting: Proactively searching for undetected threats/compromises within the network, assuming breach has occurred. Requires skilled analysts, utilizes threat intelligence, SIEM data, endpoint logs.
  • Developing Countermeasures / Deceptive Technologies: Setting traps for attackers.
    • Honeypots: Decoy systems designed to attract attackers, log their activity, distract them from real assets.
    • Honeynets: Networks of honeypots.
    • Decoy Files/Credentials: Planting fake sensitive data to detect access.
    • Simulators: Mimic real systems for analysis or deception.
  • Dynamic Network Configurations: Theoretically changing network topology to make mapping harder (less practical).
  • Security Data Analytics: Collecting, processing, analyzing security data (logs, alerts) for threat detection, incident investigation, trend analysis.
    • Processing Pipelines: Automated workflow for handling data streams.
    • Data Stream Processing: Analyzing data in real-time as it arrives.
    • Indexing and Search: Organizing log data for efficient querying (core SIEM function).
    • Log Collection and Curation: Gathering logs from diverse sources, normalizing, storing securely.
    • Database Activity Monitoring (DAM): Monitoring database access and activity for threats/policy violations.

⠀2. Preventive Processes

  • Antivirus (AV) / Anti-Malware: Detects/blocks known malicious software (signature-based, heuristics). Foundational endpoint control.
  • Immutable Systems: Infrastructure components (servers, containers) that are never modified after deployment. Updates involve replacing the entire instance. Ensures consistency, reduces configuration drift, complicates persistence for attackers. Requires robust deployment automation.
  • Hardening: Reducing the attack surface by disabling unnecessary services/ports, applying secure configurations, patching. Follow checklists/baselines (CIS, DISA STIGs).
  • Sandbox Detonation: Executing suspicious files/URLs in an isolated environment (sandbox) to observe behavior safely. Effective against zero-day malware. Can be bypassed by evasive malware (e.g., time delays - SolarWinds example).
  • Application Control: Restricting which applications can run.
    • Allow List (Whitelist): Only explicitly permitted applications can execute (more secure, harder to manage).
    • Block List (Blacklist): Explicitly forbidden applications cannot execute (less secure, easier to manage).
  • License Technologies: Enforcing software licensing (less a direct security control, more compliance/management).
  • Atomic Execution: Ensuring operations complete entirely or not at all, preventing intermediate states exploitable by race conditions (TOC/TOU problem).
  • Security Automation: Using scripts and tools to automate repetitive security tasks.
    • Scripting: Bash (Linux), PowerShell (Windows), Python (Cross-platform).
    • Scheduled Tasks: Cron (Linux), Task Scheduler (Windows). Use cases: patching, scanning, log rotation/analysis, configuration checks.

⠀3. Physical Security Reviews

  • Review of Lighting: Ensure adequate lighting around perimeter, entrances, critical areas.
  • Review of Visitor Logs: Check for completeness, accuracy, adherence to policy.
  • Camera Reviews: Verify camera placement, coverage, recording function, retention policy. Check for blind spots. Monitoring vs. just recording.
  • Open Spaces vs. Confined Spaces: Assess risks associated with different physical layouts (visibility, access control points).

⠀Exam Essentials Summary:

  • Vulnerability Management: Know the lifecycle (Discover -> Prioritize -> Assess -> Report -> Remediate -> Verify). Understand scan types (Credentialed/Non-cred, Agent/Server, Active/Passive). Know CVSS/CVE and NVD.
  • SCAP: Understand its purpose (automation/standardization) and key components (OVAL, XCCDF, CPE, CCE, CVE, CVSS, ARF).
  • Patch Management: Critical process. Know the lifecycle, importance (EternalBlue), need to test patches.
  • Information Sources: Be aware of where to get threat intel (Advisories, Bulletins, Vendors, ISACs, News).
  • Assessment Methods: Differentiate SAST (code review, no execution) vs. DAST (runtime testing) vs. Fuzzing (invalid input). Understand Wireless Scanning, SCA, Reverse Engineering, Pivoting.
  • Key Tools: Know the purpose of Nmap (port scanning), Wireshark (packet analysis), Nessus/OpenVAS (vuln scanning), Burp Suite (web proxy/testing), Metasploit (exploit framework), John the Ripper/Cain/Ophcrack (password cracking).
  • Pen Testing: Understand the methodology (Recon -> Scan -> Exploit -> Post-Exploit -> Report) and the critical need for written permission and RoE.
  • Common Vulnerabilities: Recognize major categories: Memory safety (Overflows), Input Validation (Injection, XSS, CSRF), Auth/Access Control, Crypto flaws, Misconfigurations, Outdated Components, Race Conditions. Know OWASP Top 10 context.
  • Risk Reduction Processes: Understand proactive (Hunting, Deception), preventive (AV, Hardening, Immutability, Sandboxing, App Control, Automation), and physical controls.
