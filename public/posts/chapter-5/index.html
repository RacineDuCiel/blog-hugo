Core Objective 4.2: Explain the importance of managing and mitigating vendor risk. 1. Cloud Computing Fundamentals & Shared Responsibility

  • Cloud Computing: Using a remote datacenter (CSP) for applications, data hosting, virtual servers, services consumed over the Internet. Offers on-demand/elastic scaling without physical infrastructure investment.
    • Major CSPs: AWS, GCP, Microsoft Azure, IBM Cloud.
  • Shared Responsibility Model: Fundamental concept. Confusion over this is a major risk source.
    • General Rule: CSP is responsible for security OF the cloud (infrastructure, physical security, virtualization layer). Customer is responsible for security IN the cloud (their data, configurations, access management, networking controls within their environment).
    • Responsibility Varies by Model: (See Figure 5.2 implicitly) Responsibility shifts depending on IaaS, PaaS, SaaS. Customers always retain responsibility for their data.
  • Cloud Service Models:
    • SaaS (Software as a Service): Complete packaged solution, software rented (e.g., Salesforce.com, cloud email). CSP manages almost everything.
    • PaaS (Platform as a Service): Provides a platform for development/deployment (e.g., G Suite, managed databases). Customer manages applications and data. CSP manages platform OS, infrastructure.
    • IaaS (Infrastructure as a Service): Provides basic infrastructure (compute, storage, network). Customer manages OS, middleware, applications, data. CSP manages physical infrastructure, virtualization. (e.g., AWS EC2).
  • Cloud Hosting Options:
    • Public: Available to general public (e.g., Google Drive).
    • Private: Operated for a single entity.
    • Hybrid: Mix of public and private.
    • Community: Shared among several organizations with common interests.
  • Tenancy Models:
    • Multitenancy: Single software instance serves multiple customers (isolation is key).
    • Single Tenancy: Dedicated software/instance for one customer.
  • Key Cloud Terminology:
    • Provisioning: Preparing/equipping cloud service for new users (may have startup fees).
    • Deprovisioning: Removing user/service access (may have fees).
    • Compute: Generic term for processing power, memory, networking, storage resources.
    • Elasticity: Ability to scale resources up/down on demand.

⠀2. Benefits & Risks of Cloud Computing

  • Benefits: Reduced cost (less infra/maintenance), increased storage, automation, flexibility, mobility, IT focus shifts to innovation.
  • Risks & Concerns:
    • Loss of Control: Assets outside traditional security perimeter. Trust placed in CSP.
    • Shared Responsibility Confusion: Leads to gaps in security.
    • Data Security: How is data separated/encrypted (at rest/in transit)? Who can access it?
    • Forensics/Incident Response: Chain of custody challenges, complexity in multi-tenant environments. FBI seizure example.
    • Compliance: Can CSP meet regulatory requirements (HIPAA, GDPR, PCI DSS)? Data sovereignty issues.
    • Vendor Viability: Will the CSP survive long-term? What happens if they fail/get acquired? Financial risk.
    • Vendor Lock-in/Lock-out: Difficulty moving away from a CSP due to cost, technical incompatibility, proprietary tech. Data portability issues.
    • Staff Turnover (CSP): Risk if CSP employees with access leave. Importance of CSP training.
    • Supply Chain Risks: Vulnerabilities in the hardware/software the CSP uses.
    • Cloud Outages: Impact on availability (Google Cloud example).

⠀3. Vendor Assessment & Management

  • Due Diligence: Essential before selecting a vendor/CSP. Define organizational needs first.
  • Key Assessment Questions/Areas:
    • Security: Encryption used (rest/transit)? Storage location/methods? Breach prevention? Source code escrow?
    • Infrastructure: Redundancy/backups? Supply chain visibility?
    • Network: Systems used? Patching schedule? Support availability (hours)?
    • Data: Who controls data on exit? Who has access? Employee offboarding process (vendor side)? Geographic storage location?
    • Incident Management/DR: Breach notification responsibility/process? DR plan existence/details?
    • Compliance: Confidentiality protection? Specific regulations met (GDPR, HIPAA, PCI DSS, SOX)? Right to audit?
    • Viability: Financial health? Track record? Data return policy on failure?
    • Lock-in Risk: Proprietary tech? Data/app portability?
    • SLA Terms: Uptime guarantees? Response times? Penalties? (See Agreements section).
    • Training: What security training does the vendor provide its employees?
  • Supply Chain Visibility: Understanding the components and suppliers involved in the vendor’s service/product.
  • Source Code Escrow: Agreement where source code is held by a trusted third party, released to the customer if the vendor goes out of business. Mitigates risk.
  • Ongoing Assessment: Need for periodic reviews, audits (Right to Audit clause), performance monitoring against SLAs.
  • Third-Party Dependencies: Risks associated with vendor’s use of external code, libraries, hardware.
    • Hardware: Components sourced from various manufacturers (risk shifts). IoT device risks (defaults, firmware).
    • Software/Code: Risks in open-source libraries, packages, containers (malicious code, vulnerabilities). Docker Hub example.
    • Attack Vectors: Typosquatting (malicious package misspellings), Dependency Confusion (tricking package manager with higher version numbers).

⠀4. Technical Considerations for Vendor Integration

  • Network Segmentation: Isolating vendor connections/systems from internal network (e.g., using DMZ/screened subnet). Important for compliance (PCI DSS). Reduces lateral movement risk.
  • Transmission Control: Protecting data in transit. Risk of source routing attacks (attacker modifies packet path). Mitigation: Drop/forward packets with source route option. Encryption is key.
  • Shared Credentials: HIGH RISK. Avoid sharing admin/root accounts with vendors. Use delegation where possible. Monitor vendor access. Prohibited by many policies.

⠀Core Objective 4.3: Explain compliance frameworks and legal considerations and their organizational impact. 1. Integrating Diverse Industries (Mergers, Acquisitions, Partnerships)

  • Security Risks: Inherited vulnerabilities, conflicting policies/standards, integration complexity, insider threats from merged entity.
  • Importance of Policy Review: Need for documented, up-to-date policies reflecting management view, legal requirements (SOX example), and business processes. Review questions: Changed procedures? Relevance? Legal changes? Change management integration? Best practice changes? Audit findings?
  • Stakeholder Involvement: Partner with HR, Legal, Management for compliance and policy development.
  • Change Management: Formalized process (Request -> Approve -> Review -> Test -> Schedule -> Communicate -> Implement -> Document -> Post-Review -> Rollback Plan). Controls modifications to systems/processes. ISO 20000 / NIST 800-64 definitions. Covers policy, network, system, app changes. Importance of rollback capability (“Avoid one-way doors”).

⠀2. Data Considerations

  • Data Sovereignty: Data is subject to the laws of the country where it is physically stored. Critical for compliance (esp. GDPR) and cloud provider selection.
  • Data Ownership: Defines who has rights and ultimate responsibility for data assets. Creates accountability. (vs. Data Steward: Responsible for data quality/day-to-day management).
  • Data Classifications: Categorizing data based on sensitivity (e.g., Public, Internal Only, Confidential, Restricted - from GDPR section). Determines handling controls (e.g., encryption). Ask if vendor uses classification.
  • Data Types (Examples of Sensitive Data):
    • PII (Personally Identifiable Information): Data identifying a specific individual (Name, SSN, DOB, address, email, phone, biometrics, etc.). Requires strong protection. General privacy principles apply (fair collection, purpose limitation, security, limited retention).
    • Health Information: Protected under HIPAA in the US.
    • Financial Information: Protected under PCI DSS, SOX, GLBA in the US.
    • Intellectual Property (IP): Sensitive business data.
  • Data Retention: Policies defining how long data is kept (only as long as necessary for purpose or legal requirement). HITRUST example.
  • Data Removal/Destruction/Sanitization: Securely disposing of data/media when no longer needed to prevent Data Remanence (residual data). Physical destruction is most secure. Policy requirement. HITRUST example.

⠀3. Geographic Considerations

  • Location of Data: Impacts sovereignty and applicable laws.
  • Location of Data Subject: Determines which privacy laws apply (e.g., GDPR applies to EU residents globally).
  • Location of Cloud Provider: Relevant for support, legal jurisdiction, vendor viability.
  • Geolocation Technologies: Tracking real-world location of devices/users. Uses: asset tracking, marketing. Risks: Privacy invasion, stalking.

⠀4. Third-Party Attestation of Compliance

  • Purpose: Provide assurance (to customers, regulators, stakeholders) that a vendor meets specific security/compliance standards. Verifies internal controls.
  • Methods:
    • Framework-based Assessments: Using standards like HITRUST CSF (maps to ISO, HIPAA). Can be self-assessment or external assessor.
    • Audit Reports: SAS 70 (older AICPA standard for service org controls, now SSAE 18/SOC reports). Verifies controls examined by independent auditor.
    • CSA STAR Registry: Cloud Security Alliance program for cloud provider attestation (Self-Assessment, 3rd Party Audit, Continuous Monitoring).

⠀5. Regulations, Accreditations, and Standards

  • PCI DSS (Payment Card Industry Data Security Standard):
    • Scope: Organizations handling credit/debit cards. Contractual, not law.
    • Goal: Protect cardholder data (PII).
    • Requirements: 12 core requirements (firewalls, no defaults, protect stored data, encrypt transmission, AV, secure systems, restrict access by need-to-know, unique IDs, physical access control, logging/monitoring, testing, policy).
  • GDPR (General Data Protection Regulation):
    • Scope: EU residents’ personal data, global reach. Law with large fines.
    • Goal: Strict privacy/security for personal data.
    • Key Concepts: Personal Data (PII), Processing, Data Subject, Controller, Processor. Data Subject Rights (Informed, Access, Rectification, Erasure, Restrict Processing, Portability, Object, Automated Decision-Making). Data Sovereignty impact.
  • ISO (International Organization for Standardization):
    • ISO 27002: Code of best practice for information security management (guideline, not technical). 4 categories, 93 controls (11 new).
    • ISO 27017: Additional cloud-specific security controls.
    • ISO 15408 (Common Criteria - CC): Framework for evaluating IT product security. Standardized structure/language for requirements/capabilities. Assurance via evaluation (EALs - not detailed in text).
  • CMMI (Capability Maturity Model Integration):
    • Scope: Process improvement model (originally software, now Dev, Services, Acquisition). DoD/US Gov use.
    • Goal: Increase organizational maturity. 5 Levels (Initial, Managed, Defined, Quantitatively Managed, Optimizing).
  • NIST (National Institute of Standards and Technology):
    • Scope: US Gov standards, widely adopted. Public domain.
    • NIST CSF (Cybersecurity Framework): Common language/methodology for risk management. 5 Functions (Identify, Protect, Detect, Respond, Recover).
    • NIST SP 800 Series: Special publications (guidelines, research). Examples: 800-100 (InfoSec Handbook), 800-53 (Security/Privacy Controls). EO 14028 (Software Supply Chain Security).
  • COPPA (Children’s Online Privacy Protection Act):
    • Scope: US law protecting children < 13 online.
    • Requirements: For website operators targeting children (clear policy, parental consent, parental access/deletion rights, data minimization, security, limited retention). FTC enforced, high fines.
  • CSA STAR (Security, Trust, Assurance and Risk):
    • Scope: Cloud security assurance program by Cloud Security Alliance.
    • Based on: CCM (Cloud Controls Matrix - spreadsheet mapping controls to standards).
    • STAR Registry: Public listing of CSP security posture. Levels: 1 (Self-Assessment), 2 (3rd Party Audit), 3 (Continuous Monitoring).
  • HIPAA (Health Insurance Portability and Accountability Act):
    • Scope: US healthcare providers, insurers, clearinghouses. Law.
    • Goal: Protect health insurance coverage (Title I) and standardize electronic healthcare info & protect privacy/security (Title II - Privacy Rule, Security Rule). Defines controls for PHI (Protected Health Information).
  • SOX (Sarbanes-Oxley Act):
    • Scope: US publicly traded companies’ financial records. Law.
    • Goal: Enhance corporate responsibility, combat fraud. IT impact via Section 302 (CEO/CFO certification of controls) & Section 404 (internal control assessment/reporting). Requires data archiving/protection.
  • GLBA (Gramm-Leach-Bliley Act):
    • Scope: US financial institutions. Law.
    • Goal: Protect consumer financial privacy. Requires security program, designated manager, risk assessments, policies, testing. Privacy rules (disclosure, opt-out). Prohibits pretexting.

⠀6. Legal Considerations

  • Due Care: Minimum standard of protection; actions a reasonable/prudent entity would take. Legal defense baseline.
  • Due Diligence: The process of investigation/analysis to identify risks and ensure due care is met. The activity performed. Essential for M&A, vendor selection.
  • Best Practices: Exceeding minimum requirements (due care) for enhanced security, balancing usability.
  • Export Controls: US regulations governing export of goods/tech (incl. encryption, cyber tools) to foreign entities. BIS/License Exception ACE. National security/human rights concerns.
  • Legal Holds (Litigation Holds): Formal notice to preserve ESI (Electronically Stored Information) and paper documents relevant to actual/anticipated litigation. Prevents spoliation. Applies to content & metadata. Requires process/tools. Supports E-discovery.

⠀7. Contract and Agreement Types

  • SLA (Service Level Agreement): Defines service standards (uptime, response time), remedies for failure. Often external. Uptime Agreement (UA) is a specific type (e.g., 99.999% “five nines”). Critical for cloud/vendor services.
  • MSA (Master Service Agreement): Overarching contract for long-term vendor relationship. Sets general terms (payment, warranty, IP, security). Referenced by project-specific SOWs (Statements of Work).
  • NDA (Nondisclosure Agreement): Protects confidential information shared between parties. Defines scope, obligations, duration. One-way or two-way. Common for employees/contractors.
  • MOU (Memorandum of Understanding): Less formal, documents mutual understanding/intent. Often internal or pre-contract. Avoids misunderstandings. Requires authorized signatures.
  • ISA (Interconnection Security Agreement): Specifies technical/security requirements for connecting networks/systems between organizations. Includes topology.
  • OLA (Operational Level Agreement): Internal agreement defining responsibilities of support groups to meet external SLAs.
  • PLA (Privacy Level Agreement): Specifically addresses data privacy obligations, often within cloud context.
  • BPA (Business Partnership Agreement): Formal agreement for joint ventures, defining roles/responsibilities/revenue sharing.
  • Other Related Documents:
    • BIA (Business Impact Analysis): Identifies critical functions, downtime impact, resource needs (See Ch 1).
    • IA (Interoperability Agreement): Defines requirements for data exchange between organizations.
    • COOP (Continuity of Operations Plan): Ensures business can continue during disruptions.

⠀Exam Essentials Summary:

  • Vendor Risk/Cloud: Understand the Shared Responsibility Model (CSP vs. Customer), risks (loss of control, viability, lock-in, compliance), and key assessment criteria (security, data handling, SLA, audit rights, viability). Know IaaS/PaaS/SaaS differences.
  • Compliance Frameworks: Recognize key regulations/standards and their scope: PCI DSS (Cards), GDPR (EU Data), HIPAA (US Health), SOX (US Financial Reporting), GLBA (US Financial Privacy), COPPA (US Children Online), ISO 27k (Int’l InfoSec), NIST (US Gov/Industry), CMMI (Process Maturity), CSA STAR (Cloud Assurance).
  • Legal Concepts: Differentiate Due Care (standard) vs. Due Diligence (process). Understand Legal Holds, Export Controls, Data Sovereignty.
  • Data Handling: Know concepts of Data Classification, Ownership vs. Stewardship, Retention policies, PII definition/protection, secure Destruction/Sanitization, and Data Remanence.
  • Agreements: Understand the purpose of key agreements: SLA (service levels), MSA (master terms), NDA (confidentiality), MOU (intent), ISA (interconnection), OLA (internal support).
