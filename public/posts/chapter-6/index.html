Core Objectives Covered: 1.7 (Crypto/PKI for Security), 3.5 (Implement PKI), 3.6 (Implement Crypto Protocols/Algs), 3.7 (Troubleshoot Crypto) Objective 1.7: Explain how cryptography and PKI support security objectives and requirements. 1. Definition & History

  • Cryptography: Art of protecting information by transforming it into an unreadable format.
  • Historical Examples:
    • Scytale: Wrapping message strip around a rod of fixed diameter (Spartans).
    • Caesar Cipher: Simple substitution, ROT3 (shift forward 3 letters).
    • Substitution Ciphers: Replace one character for another (e.g., Vigenère - polyalphabetic).
    • Running Key Cipher: Uses keystream with tabula recta.
    • Vernam Cipher (One-Time Pad): Theoretically unbreakable if used correctly.

⠀2. Cryptographic Goals (PAIN Acronym)

  • Privacy (Confidentiality): Keeps information secret (at rest, in transit). Achieved via Encryption.
    • Examples: Full Disk Encryption (FDE - BitLocker, CryptoForge), File Encryption (PKWARE, Sookasa), Encrypted transmission (VPN, TLS/SSL).
  • Authentication: Proves identity of sender/entity. Achieved via Digital Signatures, MACs/HMACs.
  • Integrity: Ensures information remains unaltered. Achieved via Hashing, Digital Signatures, MACs/HMACs.
    • Example: Verifying patch downloads using MD5/SHA hashes.
  • Non-repudiation: Prevents sender/receiver from denying an action. Provides proof of origin & delivery. Achieved via Digital Signatures, HMACs. Requires: Identity, Authentication, Evidence linking identity to action.

⠀3. Cryptosystem Strength Considerations

  • Factors: Algorithm strength, Key generation complexity, Key size (bits).
  • Trade-offs: Strength vs. Performance (CPU usage, time) vs. Feasibility vs. Interoperability.
  • Key Length: Larger keyspace (2^bits) makes brute-force harder. Need sufficiently sized key.

⠀4. Supporting Security Requirements

  • Compliance & Policy: Meeting guidelines, laws, regulations. Cryptography often mandated.
    • Frameworks: COBIT (auditing).
    • Regulations:
      • HIPAA: US Health info privacy/security.
      • PCI DSS: Global Cardholder data protection (contractual).
      • GLBA: US Financial institution privacy/security program.
      • SOX: US Public company financial record accuracy/controls.
      • GDPR: EU resident personal data privacy/security (global reach).
      • FERPA: US Student educational record privacy/security.
  • Privacy & Confidentiality Requirements: Protecting PII.
    • Standards/Best Practices: CIS Controls (map to NIST, PCI, HIPAA), NIST Cybersecurity Framework (Identify, Protect, Detect, Respond, Recover).
  • Integrity Requirements: Ensuring data accuracy (linked to SOX, financial scandals like Enron, Theranos). Requires proper InfoSec architecture.
  • Non-repudiation Requirements: Crucial for auditing, accounting, faceless internet transactions. Linked to access management.
  • Export Controls: Regulations (e.g., Wassenaar Arrangement) controlling export of strong crypto tech to prevent misuse.

⠀5. Common Cryptography Use Cases (Data States)

  • Data at Rest: Stored data (HDD, cloud, USB).
    • Risks: Physical theft, pod slurping, USB malware, end-of-life data recovery.
    • Protection: FDE (BitLocker, VeraCrypt), file encryption, proper disposal (wiping, zeroization, degaussing, destruction).
  • Data in Transit (Data in Motion): Data moving across networks.
    • Risks: Eavesdropping, sniffing, hijacking, alteration (due to clear text protocols like FTP, Telnet, HTTP, SMTP).
    • Protection: VPNs, HTTPS/SSL/TLS, S/MIME, OpenPGP/GPG, IPsec.
  • Data in Process/Data in Use: Data actively being used/processed in memory (RAM).
    • Risks: Unauthorized process access if OS lacks isolation.
    • Protection: Technologies like Smart Cards (often used with PIN for MFA).

⠀6. Other Common Use Cases (From Objective List)

  • Protection of Web Services: HTTPS (SSL/TLS).
  • Embedded Systems: Often use ECC due to resource constraints. Risks with embedded keys.
  • Key Escrow/Management: Securely storing keys (e.g., with a trusted party or HSM). Requires careful control.
  • Mobile Security: FDE, VPNs, specific ciphers like ChaCha (performance/battery).
  • Secure Authentication: Smart cards, EAP framework (wireless).
  • Smart Card: Protects data in use, stores keys securely, MFA.

⠀Objective 3.6: Implement the appropriate cryptographic protocols and algorithms. 1. Hashing Algorithms

  • Purpose: Provide Integrity & Authentication. Create fixed-size “fingerprint” or “message digest” from variable-size input. One-way function.
  • Collision: When two different inputs produce the same hash output (undesirable). Larger output size reduces collision probability.
  • Uses: Change monitoring (Tripwire, sfc.exe), file integrity verification (MD5sum), password storage, digital signatures.
  • Algorithms:
    • MD (Message Digest): (Ron Rivest)
      • MD2, MD4: Obsolete, vulnerable to collisions.
      • MD5: 128-bit output. Widely used for integrity checks (MD5sum), but vulnerable to collisions (not secure for signing/passwords). Processes 512-bit blocks.
    • SHA (Secure Hash Algorithm): (NIST/NSA)
      • SHA-0: Obsolete.
      • SHA-1: 160-bit output. Now considered vulnerable. Processes 512-bit blocks.
      • SHA-2: Family includes SHA-256 (strong, common), SHA-512 (stronger), SHA-224, SHA-384 (truncated versions). Recommended replacement for MD5/SHA-1.
      • SHA-3: Released 2012, uses Keccak algorithm. Different internal structure than SHA-1/2.
    • MAC (Message Authentication Code): Symmetric key used with hashing for integrity and authentication.
    • HMAC (Hash-based MAC): Uses standard hash (MD5, SHA-1/2) + shared secret key. Stronger than plain hash for authentication.
    • RIPEMD (RACE Integrity Primitives Evaluation Message Digest):
      • RIPEMD-128: Vulnerable.
      • RIPEMD-160: 160-bit output. Used in Bitcoin (alternative to SHA-256). Based on MD4.
    • Poly1305: Cryptographic MAC created by D.J. Bernstein. Fast. Used with ChaCha20. Defined in RFC 8439.
    • HAVAL: Variable output length (e.g., 128, 256 bit). HAVAL-3-128 known vulnerable.

⠀2. Symmetric Algorithms

  • Concept: Single shared key (“dual-use”) for encryption & decryption. Provides Confidentiality.
  • Pros: Fast, efficient for bulk encryption, strong with large keys.
  • Cons:
    • Key Distribution: Securely sharing the key is hard (requires out-of-band method).
    • Key Management: Number of keys needed grows quickly: n(n-1)/2. (Automation/Orchestration/PKI can help).
    • Confidentiality Only: Does not inherently provide integrity, authentication, or non-repudiation.
  • Types:
    • Block Ciphers: Encrypt fixed-size blocks of data (e.g., 64, 128 bits). Require padding if data isn’t exact multiple of block size.
    • Stream Ciphers: Encrypt data bit-by-bit or byte-by-byte. Often use keystream generator + XOR.
  • Algorithms:
    • DES (Data Encryption Standard): 64-bit block, 56-bit effective key (+8 parity bits). Obsolete, vulnerable to brute-force. Used ECB, CBC, OFB, CFB modes.
    • 3DES (Triple DES): Stopgap replacement for DES. Applies DES 3 times with 2 or 3 keys (112 or 168-bit effective key). Slow.
    • AES (Advanced Encryption Standard): Current standard. Based on Rijndael algorithm. 128-bit block size. Key sizes: 128, 192, 256 bits. Number of rounds varies (10, 12, 14). Fast, secure.
    • Blowfish: 64-bit block, variable key up to 448 bits. DES replacement. (Used by Bcrypt).
    • IDEA (International Data Encryption Algorithm): 64-bit block, 128-bit key. Used in PGP. Patented.
    • RC4 (Rivest Cipher 4): Stream cipher. Variable key (40-bit in WEP, commonly 128-bit). Known vulnerabilities (bias).
    • RC5: Block cipher. Variable block size (32/64/128), key size (0-2040), rounds (0-255).
    • RC6: Like RC5, adds integer multiplication. AES candidate.
    • ChaCha: Stream cipher (D.J. Bernstein, 2008). ChaCha20 (20 rounds) used with Poly1305 (AEAD) in TLS for mobile performance.
    • Salsa20: Stream cipher, precursor to ChaCha.
    • Others mentioned: CAST (128/256 block), SAFER (64/128 block), Skipjack (80-bit key, NSA, key escrow controversy), Twofish (128-bit block, up to 256-bit key, AES candidate).
  • Modes of Operation (For Block Ciphers): Define how multiple blocks are encrypted.
    • ECB (Electronic Codebook): Simplest. Each block encrypted independently with same key. Insecure: Identical plaintext blocks produce identical ciphertext blocks (reveals patterns). Use only for small amounts of data.
    • CBC (Cipher Block Chaining): XORs previous ciphertext block with current plaintext block before encryption. Uses an IV for first block. More secure than ECB, hides patterns. Errors propagate within block.
    • OFB (Output Feedback): Makes block cipher act like stream cipher. Uses IV + key to generate keystream, XORed with plaintext. Errors do not propagate.
    • CFB (Cipher Feedback): Makes block cipher act like stream cipher. XORs plaintext with encrypted previous ciphertext. Uses IV. Errors do propagate. Can encrypt smaller units (e.g., characters).
    • CTR (Counter): Makes block cipher act like stream cipher. Encrypts a nonce + counter value, XORs result with plaintext. Allows parallel encryption/decryption. No error propagation. Can be used with DES/AES.
    • GCM (Galois/Counter Mode): AES only. Authenticated Encryption with Associated Data (AEAD). Provides both Confidentiality and Integrity/Authentication. Uses CTR mode for encryption + hashing (GMAC) for authentication tag. Efficient, parallelizable. Common in TLS 1.2/1.3, IPsec.

⠀3. Asymmetric Algorithms (Public Key Cryptography)

  • Concept: Uses two mathematically related keys: Public Key (shared) and Private Key (secret). Based on one-way functions (trapdoors - easy one way, hard the other).
    • Math basis: Factoring large numbers (RSA), Discrete Logarithm problem (DH, ElGamal), Elliptic Curve Discrete Logarithm problem (ECC - harder).
  • Pros: Solves key distribution problem (public key shared openly), Provides Confidentiality, Integrity, Authentication, Non-repudiation, Scalable.
  • Cons: Much slower than symmetric, requires larger key sizes for equivalent strength. Best for small data (key exchange, signatures).
  • Key Agreement (Establish Shared Secret Key):
    • Diffie-Hellman (DH): Key exchange only. Allows two parties to agree on a shared secret over insecure channel. Vulnerable to On-Path (MitM) attack if unauthenticated. Parameters: p (prime), g (generator). Used in IPsec (IKE).
    • ECDH (Elliptic-Curve Diffie-Hellman): DH using elliptic curve math. Faster, allows smaller keys for same strength. Common in TLS.
  • Signing / Encryption Algorithms:
    • RSA (Rivest, Shamir, Adleman): Encryption & Signing. Based on difficulty of factoring large primes. De facto standard. Key sizes up to 3072 bits (or more). Public key (e, n), Private key (d, n), where n = p*q.
    • DSA (Digital Signature Algorithm): Signing only. Part of US Gov’s Digital Signature Standard (DSS). Uses SHA hashing.
    • ECDSA (Elliptic-Curve Digital Signature Algorithm): DSA using elliptic curves. Faster signing, smaller keys than RSA/DSA for same strength. Widely used (TLS, Bitcoin).
    • ElGamal: Encryption & Signing. Based on discrete logarithms. Extension of DH.
  • Elliptic Curve Cryptography (ECC):
    • Uses points on an elliptic curve y^2 = x^3 + ax + b.
    • Provides equivalent security with much smaller key sizes than RSA/DH (e.g., 256-bit ECC ≈ 3072-bit RSA). Efficient for mobile/embedded devices.
    • Common curves/key sizes: P-256, P-384.
  • Hybrid Encryption: Combines Symmetric + Asymmetric.
    • Process: Generate random symmetric session key -> Encrypt bulk data with session key (fast) -> Encrypt session key with recipient’s public key (secure key exchange) -> Send encrypted data + encrypted session key -> Recipient decrypts session key with their private key -> Recipient decrypts data with session key.
    • Examples: TLS/SSL, PGP, S/MIME, IPsec, SSH.

⠀4. Cryptographic Protocols

  • SSL/TLS (Secure Sockets Layer / Transport Layer Security): Secures transport layer communications (esp. HTTPS).
    • SSL is predecessor, largely deprecated due to vulnerabilities. TLS is current standard (TLS 1.2, 1.3).
    • Uses hybrid encryption: Asymmetric handshake (key exchange, authentication using certificates) + Symmetric data transfer.
    • WTLS (Wireless TLS): For WAP stack, addresses low bandwidth/power of mobile devices.
  • S/MIME (Secure/Multipurpose Internet Mail Extensions): Standard for email encryption and digital signatures. Uses PKI (X.509 certs). Built into most email clients. Provides confidentiality, integrity, auth, non-repudiation for email.
  • IPsec (Internet Protocol Security): Secures network layer (IP) communications. Built into IPv6, add-on for IPv4.
    • Components:
      • AH (Authentication Header): Provides Integrity & Authentication (hashes packet, uses HMAC). Does not encrypt.
      • ESP (Encapsulating Security Payload): Provides Confidentiality (encrypts payload). Can optionally provide integrity/auth.
    • Modes:
      • Transport Mode: Encrypts/authenticates only the IP payload. Original IP header used. (Host-to-host).
      • Tunnel Mode: Encrypts/authenticates the entire original IP packet (payload + header). Adds new outer IP header. (Site-to-site VPNs).
    • SA (Security Association): Defines parameters for the secure connection (algorithms, keys, lifetime). One-way. Uses IKE (Internet Key Exchange) (often with DH) to establish SAs and keys.
  • SSH (Secure Shell): Secure remote command-line access, tunneling, file transfer (SFTP). Replaces insecure Telnet, FTP, r-utilities. Uses encryption and authentication. Defaults to TCP port 22.
  • EAP (Extensible Authentication Protocol): Authentication framework, not a specific protocol. Used widely in wireless (WPA/WPA2/WPA3 Enterprise) and wired (802.1X). Supports various methods (EAP-TLS, EAP-TTLS, PEAP, EAP-FAST, LEAP). Often uses TLS for credential protection.
  • Other Auth Protocols (Mentioned but weak/older):
    • PAP (Password Authentication Protocol): Sends username/password in cleartext. Insecure.
    • CHAP (Challenge Handshake Authentication Protocol): Uses challenge-response with shared secret and hashing. Better than PAP, but vulnerable.
    • PPTP (Point-to-Point Tunneling Protocol): Older VPN protocol. Known vulnerabilities.
    • L2TP (Layer 2 Tunneling Protocol): VPN tunneling protocol (doesn’t encrypt itself). Often used with IPsec for encryption.

⠀5. Advanced Concepts

  • Forward Secrecy (PFS - Perfect Forward Secrecy): Ensures that compromise of a long-term key (e.g., server’s private key) does not compromise past session keys. Achieved by using ephemeral keys (e.g., Ephemeral Diffie-Hellman - DHE/ECDHE) for each session, derived independently.
  • AEAD (Authenticated Encryption with Associated Data): Encryption mode providing Confidentiality and Integrity/Authentication simultaneously. Examples: GCM, ChaCha20-Poly1305. More secure and often more efficient than encrypt-then-MAC approaches.
  • Key Stretching: Makes passwords/weak keys more resistant to brute-force attacks by adding computation time. Uses hashing + salt (random value per password) + many iterations.
    • PBKDF2 (Password-Based Key Derivation Function 2): Standard KDF. Uses HMAC + salt + configurable iterations. Can output key of desired length.
    • Bcrypt: KDF based on Blowfish cipher. Uses salt + configurable cost factor (iterations). Designed to be slow.

⠀Objective 3.5: Given a business requirement, implement the appropriate PKI solution. 1. Public Key Infrastructure (PKI) Overview

  • Purpose: Enables secure communication & transactions between parties who may not know each other, using digital certificates and public key cryptography. Provides trust framework.
  • Components: Hardware, software, policies, roles (CA, RA), certificates, CRL/OCSP.
  • Common Standard: X.509 certificates.

⠀2. PKI Hierarchy & Roles

  • CA (Certificate Authority): Trusted entity that issues, manages, validates, revokes digital certificates. (Like a passport office).
    • Root CA: Top of the trust hierarchy. Self-signed. Compromise is catastrophic. Often kept offline.
    • Subordinate CA: Issues certs to end-entities or other CAs. Gets its authority from Root CA or another Intermediate CA.
    • Intermediate CA: A subordinate CA that also issues certs to other CAs below it.
    • Trusted Providers: Commercial CAs (VeriSign/DigiCert, Thawte, Entrust, GoDaddy, Let’s Encrypt).
  • RA (Registration Authority): Verifies identity of certificate requestors before CA issues certificate. Acts as intermediary. Does not issue certs itself. Useful for large geographic deployments.

⠀3. Digital Certificates (X.509)

  • Purpose: Binds a public key to an identity (subject). Signed by CA to ensure authenticity & integrity.
  • X.509 v3 Contents: Version, Serial Number, Signature Algorithm ID, Issuer Name, Validity Period (Not Before, Not After), Subject Name, Subject Public Key Info (Algorithm, Public Key), Extensions (optional), CA Digital Signature.
  • Certificate Types:
    • Wildcard Certificate: Covers multiple subdomains under a single domain (e.g., *.example.com). Uses ***** character. Cheaper/simpler but higher risk (one key for many sites).
    • Extended Validation (EV): Requires rigorous validation of organization identity. Provides high visual trust indication in browsers (green bar - becoming less common). Strict revocation checking often required.
    • Multidomain (SAN) Certificate: Secures multiple different domain names (e.g., example.com, example.net, myothersite.org) using the Subject Alternative Name (SAN) extension. Flexible.
    • Domain Validated (DV): Basic validation, only confirms control over domain name. Quick, cheap.
    • Organization Validated (OV): Validates domain control + basic organization details. More trust than DV.
    • General Purpose: Generic term, usage defined by extensions.
  • Certificate Usages / Profiles / Templates (Key Usage / Extended Key Usage extensions): Define allowed purposes.
    • Client Authentication: Authenticates a user/device to a server.
    • Server Authentication: Authenticates a server to a client (e.g., TLS/SSL).
    • Digital Signatures: For signing data/code (non-repudiation).
    • Code Signing: Verifies software publisher identity & code integrity.
  • Extensions: Add extra information (v3 feature).
    • CN (Common Name): Traditionally the primary FQDN (e.g., www.example.com). Deprecated as primary identifier in favor of SAN.
    • SAN (Subject Alternative Name): Crucial. Lists all valid names (FQDNs, IPs) covered by the cert. Required by modern browsers. Enables multidomain certs.

⠀4. Trust Models

  • Hierarchical Model: Root CA -> Intermediate CAs -> End-entity Certs. Standard model used by public CAs. Trust flows down from Root. (Fig 6.5).
  • Single Authority Trust: One CA issues all certificates. Simple, but single point of failure/compromise. OK for small internal PKI.
  • Web of Trust: Decentralized model (used by PGP/GPG). Users sign each other’s keys based on personal verification. Trust is subjective. Lower assurance than hierarchical.
  • Cross-Certification: Creating trust relationships between different PKI hierarchies (e.g., between two companies). Allows users in one PKI to trust certs issued by the other. Enables Federation. Increases complexity & attack surface.

⠀5. PKI Life-Cycle Management & Operations

  • Key Pair Generation: Creating public/private key pair (often by client).
  • CSR (Certificate Signing Request): Formal request sent to CA containing public key and identity info, to get a certificate issued.
  • Certificate Issuance: CA verifies request (via RA), creates and signs the certificate.
  • Certificate Distribution: Making certs available (e.g., via directory, email, web server).
  • Certificate Storage: Securely storing private keys (e.g., OS store, hardware token/HSM, smart card). Tokens are hardware devices providing secure key storage & crypto operations.
  • Certificate Usage: Using cert for intended purpose (auth, signing, encryption).
  • Certificate Pinning (HPKP - HTTP Public Key Pinning): Security mechanism where server tells browser (via HTTP header) to associate specific public keys with that domain for a period. Browser will only trust certs containing one of the pinned keys. Protects against rogue CAs/mis-issued certs. Uses Trust On First Use (TOFU). Requires backup pin.
  • Certificate Stapling (OCSP Stapling): Performance/privacy enhancement for checking revocation status. Server periodically gets signed OCSP response from CA for its own cert, then “staples” (includes) this response in the TLS handshake. Avoids client needing to contact CA directly.
  • Certificate Revocation: Making a certificate invalid before its expiration date.
    • Reasons: Key compromise, info change, cessation of operation.
    • CRL (Certificate Revocation List): Periodically published, CA-signed list of serial numbers of revoked certs. Clients download/check CRL. Can have latency.
    • OCSP (Online Certificate Status Protocol): Real-time protocol for checking revocation status of a single certificate via query to OCSP Responder. Can cause latency/privacy issues if client queries CA directly.
  • Certificate Renewal / Rekeying:
    • Renewal: Extending validity period of existing certificate (and key pair).
    • Rekeying: Issuing a new certificate (with new key pair) using old identity info. Generally preferred over renewal for security.
  • Key Rotation: Policy-driven process of replacing old keys/certs with new ones before expiration or upon suspected compromise. Max validity periods are decreasing (e.g., 825 days -> potentially 398 days).
  • Key Management Issues (8 Areas): Generation, Distribution, Installation, Storage, Recovery (e.g., escrow, m of n control - requires m out of n authorized admins present), Change, Control, Disposal.
  • HSTS (HTTP Strict Transport Security): Web server HTTP header instructing browsers to only connect using HTTPS for a specified duration. Prevents SSL stripping / downgrade attacks.

⠀Objective 3.7: Given a scenario, troubleshoot issues with cryptographic implementations. 1. Common Issues & Errors (e.g., “Connection Not Private”)

  • General Approach: KISS (Keep It Simple). Check basics first (system date/time, browser cache/plugins, network connectivity) before diving deep. Use online SSL checker tools.
  • Implementation & Configuration Issues:
    • Validity Dates: Certificate expired or not yet valid. Check server/client system clocks.
    • Wrong Certificate Type: Using a single-domain cert for multiple domains, DV when OV/EV needed, wrong usage profile.
    • Revoked Certificates: Cert listed on CRL or marked revoked by OCSP. Caused by compromise, mis-issuance, etc.
    • Incorrect Name (Name Mismatch): CN or SAN entry in certificate does not match the hostname in the browser’s address bar (e.g., www.example.com vs example.com). NET::ERR_CERT_COMMON_NAME_INVALID.
    • Chain Issues / Invalid Root or Intermediate CAs: Browser cannot build a valid path from the server’s certificate up to a trusted root CA in its trust store. Intermediate certs might be missing on server, or root/intermediate is untrusted/expired.
    • Self-Signed Certificate: Certificate signed by itself, not a trusted CA. Browsers inherently distrust these unless manually imported/trusted. Chain stops at server.
    • Weak Signing Algorithm: Certificate signed using insecure hash (e.g., SHA-1). Browsers distrusting SHA-1.
    • Weak Cipher Suite: Client/Server negotiate use of outdated/insecure protocols (SSLv3, TLS 1.0/1.1) or ciphers (RC4, DES, 3DES, MD5).
    • Incorrect Permissions: OS file/store permissions prevent server/app from accessing certificate/private key.
    • Cipher Mismatches: Client and server have no mutually supported cipher suites. Connection fails.
    • Downgrade Attacks: Attacker forces client/server to negotiate a weaker protocol/cipher than they both support. HSTS helps mitigate for HTTP->HTTPS.
  • Key Issues:
    • Mismatched Keys: Private key on server does not match public key in certificate.
    • Improper Key Handling: Private key stored insecurely, accessible to unauthorized users/processes.
    • Embedded Keys: Keys hardcoded in devices/software. Difficult to update/rotate, potentially uses weak algorithms. Risk increases if device is compromised.
    • Rekeying Issues: Problems during process of generating new key pair and getting new certificate.
    • Exposed Private Keys: Private key leaked/stolen. CRITICAL. Requires immediate certificate revocation and key replacement.
    • Compromised Keys: Allows attacker to impersonate legitimate entity, decrypt traffic (if not using PFS), forge signatures, perform MitM attacks. Requires revocation.
  • Key/Data Destruction Concepts:
    • Crypto Shredding: Intentionally deleting the encryption key to make data permanently inaccessible (theoretically - brute force might still work eventually). Used for data disposal requirements. Requires unique keys per data set.
    • Cryptographic Obfuscation: Masking/scrambling data value, making it hard to understand/recover, but not true encryption. Can be reversible. Different from “security through obscurity”.

⠀Exam Essentials Summary:

  • Goals: Understand PAIN (Privacy, Authentication, Integrity, Non-repudiation) and how crypto achieves them (Encryption, Hashing, Signatures).
  • Types: Differentiate Symmetric (single shared key, fast, bulk, confidentiality, key distribution issue) vs. Asymmetric (public/private keys, slow, key exchange/signatures, CIA+N) vs. Hashing (one-way, integrity). Know Hybrid approach.
  • Algorithms: Recognize key algorithms & their purpose/strength: AES (Symmetric standard), RSA/ECC (Asymmetric standards), SHA-2/3 (Hashing standards), DH/ECDH (Key exchange), DSA/ECDSA (Signing), GCM/ChaCha20-Poly1305 (AEAD).
  • PKI: Understand components (CA, RA, Certs, CRL/OCSP), hierarchy (Root, Intermediate, Subordinate), trust models (Hierarchical, Web), certificate types (Wildcard, SAN, EV), lifecycle (CSR, issuance, revocation), and key concepts (Pinning, Stapling, HSTS).
  • Protocols: Know where crypto is applied: TLS/SSL (Transport), IPsec (Network), SSH (Application), S/MIME (Email), EAP (Authentication).
  • Use Cases: Apply crypto concepts to Data at Rest (FDE), Data in Transit (VPN, TLS), Data in Use (Smart Cards), Code Signing, Web Services, Mobile, etc.
  • Troubleshooting: Recognize common certificate errors (validity, name mismatch, chain issues, revocation) and key management problems (compromise, weak ciphers/keys).
