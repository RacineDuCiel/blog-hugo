Core Objective 2.7: Given an incident, implement the appropriate response. 1. Foundational Concepts & Importance

  • Need for Speed: Rapid response is critical, especially against advanced actors (per Mandiant CEO testimony re: SolarWinds).
  • Maturity Matters: Org security maturity dictates IR capability (Untested Plan -> Mature SOC/Threat Hunting). Many orgs are reactive (notified by 3rd parties).
  • Dwell Time: Time attackers remain undetected; stresses analysts for real-time response.
  • CASP+ Role: Must conduct IR quickly, efficiently, follow documented recovery procedures.
  • Design for Response: Systems should facilitate IR (roots in Morris worm -> CSIRT creation).

⠀2. Event Classifications & Triage

  • Event: An observed occurrence; potentially malicious, potentially benign (e.g., admin password change alert).
  • Triage Event: Process of determining if an event is a true incident or a false positive. Prioritizes investigation efforts.
  • Incident: An event with a confirmed negative consequence (e.g., system breach, outage, data loss/destruction). Requires IR playbook activation. (Example: Password change + data exfiltration to known malicious IP).
  • Crisis: An incident escalating out of control, requiring immediate attention due to significant business impact or threat of public disclosure. (Example: Exfiltration of critical IP from classified server by terminated admin + ransom demand).
  • Classification Accuracy: Critical to avoid wrong conclusions.
    • False Positive (FP): Determined true when it’s false (False Alarm). Security Preference: Prefer FP (metal detector alarms) over FN for high risk.
    • False Negative (FN): Determined false when it’s true (Missed detection). Security Preference: Huge risk if threat missed (gun gets through detector). Court Preference: Prefer FN (guilty found innocent) over FP (innocent convicted).
    • True Positive (TP): Correctly identified as true/malicious.
    • True Negative (TN): Correctly identified as false/benign.

⠀3. Pre-Escalation Tasks & Preparation (Step 1 of IR Process)

  • Understand Attack Vectors: Know likely threats (NIST examples: removable media, phishing, web attacks, lost/stolen equip).
  • Develop Specific Playbooks: Different responses needed for different incidents (e.g., lost phone playbook includes encryption status, remote wipe procedure).
  • Establish Policies & Procedures: Foundation for IR activities. Define CSIRT roles.
  • Address Common Violations: Plan should cover DoS, ransomware, exfiltration, social engineering, theft, malware, data breach, terrorism, unauthorized access/software, physical intrusion, web attacks, viruses/worms.
  • Incorporate Threat Intelligence: Prioritize responses based on likely actors targeting the industry/org (e.g., APT35 targeting US Gov/Energy/Telecom via spear phishing, specific malware like DRUBOT). Scan for specific TTPs/IoCs.

⠀4. The Incident Response Process (PICERL + Lessons Learned)

  • Goal: Reduce damage, maintain/restore business continuity, defend future attacks, enable legal action.
  • CSIRT (Computer Security Incident Response Team): Team executing the plan.
  • IRP (Incident Response Plan): The documented plan/playbook.
  • Steps: 1 Preparation: Policies, procedures, tools, training, threat intelligence (Covered above). 2 Detection (Identification): Identifying events/incidents. Use automated tools: IDS/IPS, firewalls, logs, SIEM (centralized log analysis/anomaly detection). 3 Analysis: Evaluating incident nature, specifics, threat level. Verify if real (not FP). Use detection tools + expertise. 4 Containment: Stopping the incident from spreading/worsening. Execute plan steps. Requires trained team. Includes removing the problem. 5 Recovery (Eradication & Recovery): Restoring affected systems to full functionality safely and quickly. Restore data from backups, re-establish settings per plan. 6 Investigation & Closure (Post-Incident Activity): Determine what/who/how. Prepare report (formal/informal) for policy evaluation. (Text slightly differs from standard PICERL by including Investigation here). 7 Lessons Learned (Post-Incident Activity): Review what happened, successes, failures, necessary changes to prevent recurrence. Document in AAR (After-Action Report).

⠀5. Response Playbooks & Processes

  • Playbook: Overarching response strategy for a specific incident type (e.g., ransomware, phishing). May incorporate multiple runbooks.
    • Components: Triggering condition, major steps (policy/procedure), best practices, end goal/desired state, compliance addressed.
    • Building: Identify triggers -> Map possible actions -> Categorize required actions -> Define process order -> Define end state/compliance.
    • Resources: MITRE ATT&CK (TTPs), Vendor examples (Rapid7 ransomware playbook), incidentresponse.com.
  • Runbook: Detailed procedures for specific, routine tasks/processes. A “chapter” within a playbook. Can be automated.
  • Response Methods:
    • Non-automated: Human intervention/decision-making.
    • Automated: Crucial for speed.
      • Automated Runbooks: Translate Ops/Sec knowledge into procedures run automatically based on criteria. Reduces wait times.
      • SOAR (Security Orchestration, Automation, and Response): Advanced tooling. Collects threat data from multiple sources, automates responses. Integrates tools, provides pre-built actions. Increases efficiency, consistency; combats alert fatigue/skill gaps. Requires proper configuration/customization.

⠀6. Communication Plan & Stakeholder Management

  • Stakeholders (CSIRT members): Diverse skills needed. InfoSec, Legal, Admins (Network/System), Physical Security, PR, HR.
  • Communication Plan:
    • Crucial for managing media/public perception.
    • Designate ONE spokesperson.
    • Communicate only verified facts: what happened, customer impact, remediation steps.
    • Forward all media/stockholder questions to spokesperson. Avoid speculation.
  • Stakeholder Management (Senior Management):
    • Immediate question: “How bad is it?”
    • Factors: Scope (systems/data affected), Impact (effort/time to recover), Cost (monetary, downtime), Legal Ramifications (data disclosure, regulations).
    • Answers require full identification/containment first. Legal counsel involvement needed for damage control.

⠀7. Hunt Teaming (Proactive Detection)

  • Definition: In-house, proactive search for vulnerabilities/compromises before an incident forces reaction.
  • Methods: Log analysis (SIEM helps), Heuristics, Behavioral analysis (leverages security pro “gut feeling”).
  • Overlap with Vuln Management: Finds OS, Application, Firmware vulnerabilities.
  • Mitigation: Host-based firewalls/IDS help vs. OS vulns. Patching helps vs. local exploits. Firmware vulns harder to exploit without local access (less risk in secured enclaves).
  • Secured Encrypted Enclaves: Separate hardware coprocessors enhancing device security.

⠀Core Objective 2.8: Explain the importance of forensic concepts. 1. Definition & Purpose

  • Digital Forensics: Structured analysis of digital data in a legally approved way for court-admissible evidence.
  • Categories:
    • Computer Forensics: Examining physical media.
    • Software Analysis: Reviewing malware/code.
    • Network Analysis: Analyzing traffic/logs.
  • Legal vs. Internal: Forensic process must always meet legal standards, regardless of internal corporate needs. Legal requirements often stricter.

⠀2. Principles, Standards & Practices (IOCE)

  • IOCE Principles (International Organization on Computer Evidence): 1 Apply general forensic principles. 2 Actions must not change original digital evidence. 3 Only trained personnel should access original evidence when necessary. 4 Fully document all activities (seizure, access, storage, transfer). 5 Individual handler is responsible for evidence in their possession. 6 Responsible agency must ensure compliance.
  • Best Practices (Handling Media): 1 Secure original (unchanged state). 2 Hash original. 3 Make two bit-level copies (exact duplicates incl. hidden/slack space). 4 Hash both copies upon creation (verify exactness). 5 Secure original + primary copy (library/control). 6 Use second copy (working image) for analysis.

⠀3. The Forensic Process

  • Goal: Discover what occurred, decide sanctions, prepare court-admissible evidence.
  • General Steps: 1 Identification & Acquisition: Identify relevant info, retrieve it. Create bit-level copy. Original remains intact. 2 Preservation & Authentication: Maintain evidence in unchanged state. Use hashing (MD5 - still used for integrity checks, SHA-256 - NIST recommended) to prove integrity. Document everything. 3 Analysis, Recording & Presentation: Examine working copy carefully (slack space, hidden files, cache, logs). Document all activities. Extract findings. Present results.
  • E-discovery (Electronic Discovery): Process of finding, extracting, reviewing electronic info (structured & unstructured data like emails, chats) for legal cases. Governed by legal rules.

⠀4. Key Forensic Concepts

  • Chain of Custody: Critical for evidence reliability/credibility. Continuous, documented log answering: Who collected/possessed? Where collected/stored? How collected/stored? Why accessed/moved? Starts at seizure, ends at disposal/return.
  • Order of Volatility (IETF RFC 3227): Collect evidence from most volatile to least volatile components. 1 Registers, Cache 2 Routing table, ARP cache, Process table, Kernel stats, Memory (RAM) 3 Temporary file systems 4 Disk (HDD, SSD) 5 Remote logging/monitoring data 6 Physical configuration, Network topology 7 Archival media (backups)
  • Evidence Preservation: Secure storage, backups, preventing tampering/contamination. Evidence is fragile.
  • Integrity Preservation: Using hashing (MD5, SHA-256) to prove data hasn’t changed since collection.
  • Legal Holds (Litigation Holds): Formal process to preserve ESI (Electronically Stored Information) & paper documents when litigation is anticipated. Prevents spoliation (destruction/alteration of evidence). Covers content & metadata. Requires process/tools. Supports e-discovery.
  • Data Ownership: Legal rights holder over data; influences protection decisions.
  • Data Recovery & Storage: Relates to backups (RAID, shadowing), DR, compliance needs.
  • Electronic Inventory & Asset Identification: Knowing what assets exist and their value is crucial for forensics (identifying compromised assets). Asset management foundation.
  • Data Retention Policies: Dictate how long data is kept (influences e-discovery scope). (IRS 7-year example).
  • Systems Designed for Incident Response: Build security/logging/response capabilities in from the start (SDLC involvement). Cheaper than adding later. (NIST approach: Implementation, Monitoring, Compliance, Strategy, Policy, Awareness).

⠀5. Related Analysis Concepts

  • Cryptanalysis: Breaking codes/ciphers without the key. Finds weaknesses. Work Factor: Effort needed. Tools: CrypTool, EverCrack, CryptoBench.
  • Steganalysis: Detecting/recovering hidden data (steganography). Techniques: Stego-Only (only stego file), Known-Stego (original + stego file), Known-Message (hidden message + stego file). Made harder by encryption.

⠀Core Objective 2.9: Given a scenario, use forensic analysis tools. 1. Tool Categories & Purpose

  • Variety of tools used; common command-line utils often supplement specialized tools.

⠀2. File Carving Tools

  • Definition: Selectively extracting structured data (files) from raw data (often unallocated disk space) by analyzing headers/footers.
  • Foremost: Linux-based recovery tool (images, iPhones). Reads drive portions into memory. Free, originally for law enforcement.
  • Strings: Used by tools like Scalpel (open-source carving tool) to find headers/footers. Scalpel uses algorithms like Boyer-Moore for efficient pattern searching.

⠀3. Binary Analysis Tools

  • Definition: Threat/vulnerability analysis at the binary code level (when source code unavailable).
  • Hex Dump (hexdump): Linux utility (or Windows freeware) to view file contents in hex/decimal/octal/ASCII at byte level.
  • Binwalk: Analyzes/extracts firmware images (IoT). Uses libmagic library.
  • Ghidra: NSA reverse engineering tool (decompiler) for malware analysis. Free alternative to IDA-Pro.
  • GDB (GNU Project Debugger): Linux/Unix debugger. Finds bugs, analyzes crashes (core dumps).
  • OllyDbg: Windows 32-bit assembler level debugger. Malware analysis, attach to running processes.
  • Objdump: Linux command; displays info about object files.
  • readelf: Linux command; displays detailed info about ELF files (executables, libraries). Doesn’t use BFD library (unlike objdump).
  • Strace: Linux diagnostic tool; monitors process-kernel interactions (system calls, state changes). Debugging without source code.
  • file: Linux command; identifies file type (filesystem, magic, language tests). Starting point for binary analysis.

⠀4. Analysis Tools

  • Purpose: General analysis, threat intelligence, reverse engineering.
  • VM Distributions: REMnux (Linux, malware analysis), Mandiant Threat Pursuit VM (Windows, TI tools: OpenCTI, Splunk, Maltego, Caldera).
  • ExifTool: Reads/writes/edits metadata in images, audio, video, PDF files. Reveals info like camera model, date/time, owner, GPS.
  • Nmap: Network scanner. Finds live hosts, open ports, services, OS versions. Command-line (Nmap) & GUI (Zenmap).
  • Aircrack-ng: Wireless security toolkit (monitoring, attacking, testing, cracking WEP/WPA/WPA2). Command-line, in Kali. Less effective vs. WPA3 due to Dragonfly handshake (resists offline dictionary attacks).
  • Volatility: Open-source memory forensics tool (Python). Analyzes RAM dumps (live memory captures) for running processes, network connections, command history, clipboard data. Plugin support. Alternatives: SANS SIFT, Redline.
  • The Sleuth Kit (TSK): Command-line tools for disk image analysis (find/recover files). Used by Autopsy (GUI frontend). Commands like blkls (display data blocks).
  • Dynamically vs. Statically Linked Libraries:
    • Static: All library code copied into final executable. Larger file, independent.
    • Dynamic: Libraries linked at runtime by OS. Smaller file, dependencies required. Affects reverse engineering approach.

⠀5. Imaging Tools

  • Purpose: Create bit-for-bit forensic copies of drives/devices.
  • FTK Imager: Free tool (AccessData). Creates images (raw, E01, AFF, SMART formats). Can image physical/logical drives, files/folders, capture live memory.
  • dd: Standard Linux/Unix command-line utility. Copies data block-by-block. Creates images (ISO), backups (MBR), converts formats. Can calculate hashes (MD5, SHA). Powerful, free. Can be used from bootable Linux distros (e.g., Trinux).
  • Clonezilla: Imaging/cloning software (Live for single machine, SE for server/enterprise). Can create/restore images. Used with bootable USB (Rufus helps create).

⠀6. Hashing Utilities

  • Purpose: Verify data integrity and authenticity.
  • sha256sum: Linux utility computes SHA-256 hash. Verifies integrity.
  • ssdeep: Computes fuzzy hashes (CTPH - context-triggered piecewise hashes). Can find similar/modified files, even with deletions/insertions. Fast identification. In Kali.

⠀7. Live Collection vs. Post-Mortem Tools

  • Live Collection: Gathering volatile data from a running system.
  • Post-Mortem: Analyzing data after system shutdown (disk images, logs).
  • Tools:
    • nbtstat: Windows command; finds NetBIOS names associated with IP addresses.
    • netstat: Displays network connections (TCP/UDP, listening ports, established sessions), routing tables, protocol stats. -an (all, numeric), -sp (stats per protocol). Useful for identifying active connections on compromised host. Understand ports (0-65535), TCP vs. UDP, well-known (0-1023), registered (1024-49151), dynamic (49152+).
    • ps: Linux/Unix command; shows current process status (PID, TTY, TIME, CMD). Used to identify/kill processes.
    • vmstat: Linux command; reports virtual memory statistics (processes, memory, paging, CPU activity).
    • ldd: Linux command; shows shared library dependencies of an executable.
    • lsof: Linux command; lists open files and the processes using them.
    • conntrack: Linux command; manages connection tracking subsystem (list/monitor connections).
    • memdump: Dumps memory contents for a specific process (or kernel - risk of system lock).
    • tcpdump: Command-line packet analyzer. Captures/displays network traffic.
    • Wireshark: GUI packet analyzer/sniffer. Captures traffic (.pcap files), analyzes packets layer by layer (OSI). Powerful for troubleshooting and attack analysis. Requires permission due to potential eavesdropping. Needs supported NIC.
    • Netcat (nc): Networking utility (“Swiss Army knife”). Reads/writes data across network connections (TCP/UDP). Versatile backend tool.

⠀Exam Essentials Summary:

  • Incidents vs. Events: Understand the difference (Event=observed, Incident=confirmed negative impact, Crisis=escalated incident). Know FP/FN implications. Triage is key.
  • IR Process: Know the steps (Prep, Detect, Analyze, Contain, Recover, Investigate/Close, Lessons Learned). Understand the role of CSIRT, IRP.
  • Playbooks/Runbooks/SOAR: Playbooks guide response to specific incident types (Ransomware, Exfil, Social Eng.). Runbooks detail specific tasks. SOAR provides automation/orchestration.
  • Forensic Concepts: Legality is paramount. Follow IOCE principles. Master Chain of Custody & Order of Volatility. Understand Hashing for Integrity. Know Legal Holds.
  • Forensic Tools: Recognize key tools by category/purpose: File Carving (Foremost, strings/Scalpel), Binary Analysis (Ghidra, debuggers, hex viewers), Analysis (Volatility, Nmap, Wireshark, TSK), Imaging (FTK Imager, dd), Hashing (sha256sum, ssdeep), Live Collection (netstat, ps, lsof, tcpdump).
