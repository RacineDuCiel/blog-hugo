#certifications/CASP

Overall Goal: Implement enterprise security as a holistic framework, translating business requirements into executable security requirements using principles, methods, frameworks, and modeling for defense in depth. Move beyond basic perimeter security and antivirus. Objective 1.1: Secure Network Architecture Requirements & Objectives A. Foundational Concepts

  • Need for Architecture: Build a strong foundation to reduce risk and protect assets.
  • Hardening: Make attacker’s job difficult (remove unneeded services/ports, secure configurations, least privilege). Secure root/admin accounts are critical. Default accounts are a huge risk.
  • TCP/IP Weaknesses: Designed for usability, not security. Many protocols lack encryption (clear text). Weaknesses in protocols or software implementations.
  • Strategy: Balance needs, requirements, options, cost. Accept documented concessions based on risk/likelihood. Plan for resilience (e.g., backup facilities outside hurricane zones).

⠀B. Key Services & Technologies 1 Load Balancer: * Purpose: Makes multiple systems appear as one resource. Provides redundancy, increased capacity (distributes load), allows rolling upgrades, centralized defense. * Proxy Load Balancer: Acts as reverse proxy, distributes traffic. * Layer 4: Uses Network/Transport layer info (IP, TCP, UDP). * Layer 7: Uses Application layer info (HTTP). * Function: Monitors health, sends requests only to responsive servers. Improves performance by offloading session management. 2 IDS/IPS (NIDS/HIDS/WIDS/WIPS): (NIST term: IDPS) * IDS (Intrusion Detection System): Gathers/analyzes info to detect intrusions. Passive. * IPS (Intrusion Prevention System): Builds on IDS; can react automatically to prevent intrusions. Active. * Detection Methods: * Signature Recognition (Misuse Detection): Matches activity against database of known attacks. Needs updates. Cannot detect unknown/polymorphic/encrypted attacks easily. (e.g., Snort). * Anomaly Detection (Heuristics/Behavior): Compares activity to a baseline of normal behavior. Can detect unknowns but may miss slow changes. * Protocol Decoding: Understands protocol specifications (TCP/IP stack) to find abnormal activity. Needs state information. Effective vs. custom stacks. * IDS Components: Sensors (collect), Central Monitor (analyze), Report Analysis (counteract), DB/Storage (trend analysis), Response Box (block/drop/redirect), Alert Definitions (rules), Alerts. * Snort Rules: Actions (alert, log, pass, activate, dynamic, drop, reject, sdrop). Custom rule types possible. Options (General, Payload, Nonpayload, Postdetection). * Thresholds: Fixed (numeric calculation), State-Based (change in metric state), Historical (compare current vs. past). * Alert Fatigue: Overwhelming number of alerts makes distinguishing true positives difficult. Tiered SOC response helps. Aggregation may be needed. * Sensor Placement: External (DMZ/Screened Subnet), Internal, Specific hosts, Inline (series). Determines detection scope. Needs tuning/baselining. * NIDS (Network IDS): Captures/analyzes network traffic. Inspects packets. Layer of defense. Cons: Blind to encrypted traffic (unless decrypted elsewhere), vulnerable to insertion/splicing/fragmentation attacks, high load issues. * HIDS (Host IDS): Monitors a specific computer system. Compares host activity to baseline. Resides on host. Pros: Can analyze decrypted traffic. Cons: Consumes host resources. * WIDS/WIPS (Wireless IDS/IPS): Focus on detecting/preventing rogue wireless devices. WIPS can actively block. Manual review recommended before blocking (avoid impacting neighbors). * INE/IME (Inline Network/Media Encryptor): High-security devices for strong (Type 1) encryption over networks (INE) or between CPU/disk (IME). Used in high-security environments. INE differs from VPN (not point-to-point). 3 WAF (Web Application Firewall): * Purpose: Protects web applications (not a replacement for traditional FW). Sits between client and web server. Analyzes Layer 7 traffic (HTTP, HTTPS, SOAP, XML). * Capabilities: Protects against XSS, SQLi, hidden field tampering, cookie poisoning. Deep packet inspection. * Detection: Signature-based or anomaly-based (vs. normal traffic patterns). * Tools: Examples: ModSecurity (open-source), Barracuda, Fortinet, Cisco. WAF detection/bypass tools exist (w3af, wafw00f). 4 NAC (Network Access Control): * Purpose: Verify device health/compliance before allowing network connection. Prevents/quarantines non-compliant devices. * Types: * Infrastructure-based: Requires hardware/OS upgrades (e.g., Cisco, Win 11). * Endpoint-based: Software agent on clients, managed centrally. * Hardware-based: Network appliance monitors behavior. * Agentless NAC: Policy enforcement integrated with authentication (e.g., AD); applied at logon/logoff. 5 VPN (Virtual Private Network): * Purpose: Secure, reliable transport over insecure networks (Internet). Encrypts data between endpoints. * Types: * Hardware-based: Dedicated appliance handles crypto. Good for site-to-site. Adds complexity/cost. * Software-based: Easy to implement. Relies on CPU. Good for smaller firms. May rely on 3rd party providers (control issue). * Protocols: * PPTP: Older, based on PPP. * IPsec: Network Layer (L3). Modes: Tunnel (entire packet encrypted, new header) vs. Transport (payload only encrypted, original header). Common for traditional VPNs. Requires client software. * SSL/TLS: Application Layer (L7). (Referred to as SSL VPN, but uses TLS). Common in browsers. Allows stricter, per-application access control. Simpler user setup, lower cost (no special client needed). * Access/Authentication (See also 1.5): * Centralized: Single entity verifies (e.g., RADIUS, Diameter). * Decentralized: Verification throughout system. * Protocols: RADIUS (older, modem pools -> 802.1X/EAP), Diameter (RADIUS successor, IP mobility), PAP (clear text, insecure), CHAP (challenge-response, shared secret, better than PAP), MS-CHAPv2 (PPTP auth). LDAP/AD also used. * Placement: Parallel to Firewall (scalable, no central inspection), In Screened Subnet (FW inspects decrypted traffic, potential bottleneck), Integrated VPN/Firewall (easier mgmt, single point of failure risk unless mirrored). 6 DNSSEC (Domain Name System Security Extensions): * Purpose: Provide integrity and authentication for DNS responses (server identity). Digitally signs responses. * Requirements: Needs pre-distribution of authentication keys. * Limitations: Does NOT provide confidentiality. Does NOT protect against DDoS or domain hijacking. Authenticates server, not content (can still sign impersonated domain). * Context: Secures DNS beyond basic methods (internal/external servers, secure zone transfers - TCP 53). Addresses DNS vulnerabilities (cache poisoning, DoS). BIND alternatives (Unbound). 7 Firewall/UTM/NGFW: * Firewall (General): Hardware/software at boundary (trusted/untrusted). Uses rules (ACLs) for traffic control. * UTM (Unified Threat Management): All-in-one appliance (Firewall, NIPS, Gateway AV/Anti-spam, VPN, Content Filtering, Logging). Pros: Simplicity, single management console. Cons: Single point of failure. Placement: Network edge. * NGFW (Next-Generation Firewall): (Implied) More advanced firewall capabilities beyond basic ACLs (likely includes features found in UTM/IPS/WAF). 8 NAT Gateway / Internet Gateway: * NAT History: Addresses IPv4 exhaustion. Uses private IP ranges (RFC 1918: 10.0/8, 172.16/12, 192.168/16). * NAT Gateway (NGW): Enables devices in private subnet (no public IP) to connect outbound to Internet/cloud. Prevents inbound connections to those devices. * Internet Gateway (IGW): Allows instances with public IPs to connect to the Internet. 9 Forward/Transparent Proxy: * Definition: Server intercepts user connection to Internet. Does not modify requests/responses (transparent). * Purpose: Monitor user behavior, restrict access (content filtering), block malicious traffic. * Example: Squid. 10 Reverse Proxy: * Definition: Server sits at network edge. Client connects to proxy, proxy connects to origin server(s). * Purpose: Policy management, traffic routing, load balancing, scrubbing traffic (DDoS protection, WAF functionality), rerouting for availability. 11 DDoS Protection: * General: Mitigate crippling effects of DDoS attacks (service degradation, bandwidth/CPU exhaustion). * Techniques: Filtering (Reverse Proxy, WAF), Black Holing (RTBH filtering via routing protocols - send to Null0), ISP-level mitigation. CLDAP reflection example. 12 Routers: * Function: Layer 3 device, forwards packets based on destination IP using routing tables built by routing protocols (e.g., RIP, OSPF). Handles routable protocols (e.g., IP). * Security Risks: Misconfiguration, route poisoning, ICMP redirect attacks. * Security Controls: * Transport Security: IPsec, TLS/SSL for management/data. * Port Security (Switch feature): Layer 2 control. Limits number of source MACs per port. Deters adding unauthorized switches. * RTBH Filtering: Drop unwanted traffic (e.g., DDoS) using routing updates (send to Null0). * Trunking Security (Switch feature): Secure links carrying multiple VLANs (trunks - 802.1Q, ISL). Prevent VLAN hopping (attacker jumps VLANs). Mitigation: Disable DTP on user ports. * Route Protection: Ensure end-to-end network flow security (method depends on network type). 13 Mail Security: * Vulnerabilities: Clear text transmission, spoofing, spam, phishing, malware delivery. * Protocols: SMTP (send/relay, TCP 25 / Secure 587/465), POP3 (retrieve, TCP 110 / Secure 995), IMAP (retrieve, TCP 143 / Secure 993). Secure versions use SSL/TLS. * Protection: Encryption (PGP, S/MIME, SSL/TLS), Gateway Anti-spam/Antivirus (UTM feature). 14 API Gateway / XML Gateway: * API (Application Programming Interface): Interface definition for client/server communication. Vulnerable if not secured. * API Gateway: Sits between client & backend service. Validates requests, routes traffic, acts as proxy, enforces policies (auth, rate limiting, input filtering). * XML Gateway: Entry point/proxy specifically for web services using XML messages. 15 Traffic Mirroring: * Cloud Context: Copies inbound/outbound traffic from cloud instances/LBs. * Purpose: Send copy to monitoring tools for content inspection, threat monitoring, troubleshooting. 16 SPAN Port / Port Mirroring: * Switch Context: Copies packets from specified switch port(s) to a dedicated monitoring port. * Purpose: Network analysis, troubleshooting (e.g., for NIDS, sniffers). 17 VPC (Virtual Private Cloud): * Definition: Logically isolated private cloud environment hosted within a public cloud. * Purpose: Enhanced security, granular control, customizable network environment. 18 Network Tap: * Definition: Hardware device inserted inline to passively copy traffic. Typically 4 ports (2 network inline, 2 monitor out). * Purpose: Monitoring, testing, troubleshooting without disrupting traffic flow. 19 Sensors: * Definition: Collects information (e.g., firewall log, network tap). * Purpose: Feed data analysis tools for defense decisions. * Types: Host-based (accurate, local), Network-based (broad coverage, potentially vulnerable). Redundancy needed. 20 SIEM (Security Information and Event Management): * Purpose: Centralized collection, analysis, correlation, reporting, storage of logs/events from multiple sources. Helps identify threats, manage alert fatigue. * Combines: SIM (long-term storage) + SEM (real-time reporting). * Features: Alerting, aggregation, review, storage, visualization, pattern monitoring. * Challenges: Complex setup, requires multiple databases. * Vendors: ArcSight, Splunk, Lacework, Exabeam. 21 FIM (File Integrity Monitoring): * Purpose: Detect unauthorized changes to files. * Modes: * Agent-based: Real-time monitoring, uses host resources, efficient (only reports changes after baseline). Needs agent maintenance. * Agentless: Scheduled scans, no real-time detection, rescans entire system each time. Easier operation (no agents). 22 SNMP Traps: * SNMP Overview: Network management protocol (UDP 161/162). Agent reports stats to Manager. v1/v2 insecure (clear text community strings). v3 adds encryption. * Trap: Asynchronous alert message sent from Agent to Manager upon specific event (e.g., overheating, error condition). Used for monitoring. 23 NetFlow: * Purpose: Network flow analysis tool (orig. Cisco). Collects traffic metadata (source/dest IP/port, protocol, timestamps, volume). Helps identify anomalies (exfiltration, misconfigs, odd behavior) in near real time. Provides different perspective than packet capture. * IPFIX: IETF standard alternative to NetFlow. 24 DLP (Data Loss Prevention): * Purpose: Prevent sensitive data exfiltration. Analyze egress traffic for anomalies. * Requires: Data flow diagram for placement (often boundary device with deep packet inspection). Policies defining sensitive data. 25 Antivirus: * Purpose: Detect/eradicate malware. Essential first defense. * Detection: Signature-based (needs updates) vs. Heuristic/Behavioral (detects unknowns, uses AI/patterns). * Virus Techniques: Boot Record, Macro, Program, Multipartite infectors. Fast vs. Sparse infection. RAM-resident. Polymorphic (changes signature). * Mitigation: Keep AV updated (workstations, servers, mobile). End-user awareness training (attachments, unknown programs). Layered defense (zones). EICAR test file for verification. * Examples: Bitdefender, Trend Micro, Norton, McAfee.

⠀C. Segmentation Strategies 1 Segmentation (General): Place systems with different functions/security levels in different zones. Use security, network, or physical boundaries. Limits attacker lateral movement. Used in IR to isolate systems. 2 Microsegmentation: Fine-grained segmentation within datacenters, allowing security controls per unique segment. Application-level security. 3 LAN/VLAN: LAN (local network). VLAN (logical segmentation on switches based on port assignment). Benefits: Reduced congestion, smaller broadcast domains, security separation. Risks: VLAN hopping (mitigate by disabling DTP). 4 Jump Box/Host/Server: Hardened system used as secure origin point for administrators to access systems in different/less trusted zones. Protects admin credentials. Evolving to SAW (Secure Admin Workstation). Never use for non-admin tasks. 5 Screened Subnet (DMZ): Perimeter network between two firewalls (or using triple-homed firewall). Hosts public-facing services (web, email, DNS). Isolates internal network from direct Internet access. 6 Data Zones: Creating layers of protection based on data sensitivity/location. * Vector-Oriented: Focus on attack vectors (e.g., disable USB). * Info-Centric: Layer controls on data itself (info, app, host, network controls). * Protected Enclaves: Define critical areas with stronger controls (VPNs, FWs, VLANs, restricted access). 7 Staging Environments: Test environment closely matching production. Used to test patches/updates before deployment to avoid breaking changes or introducing vulnerabilities. Note: Attackers may use delays to bypass staging detection (SolarWinds). 8 Guest Environments: Separate network segment for visitors/unmanaged devices. Provides Internet access only (typically). Requires Terms of Use, security controls (MFA, timeouts, classification, permission reviews if sensitive access needed). 9 VPC/VNET: Cloud-based segmentation. VPC (isolated section of public cloud). VNET (software-defined virtual network). Provides customer control over IP addressing, subnets, routing, firewall rules within the cloud. 10 Availability Zone: Physically separate datacenters within a geographic region. Used for high availability and redundancy/failover. 11 NAC Lists (ACLs): Firewall/router rules defining allowed/denied traffic based on criteria (IP, port, protocol, etc.). Processed top-down, implicit deny at end. Standard (source IP only) vs. Extended (more criteria). Basic data flow enforcement. 12 Policies/Security Groups: Policies (high-level docs from management). Security Groups (Cloud term, e.g., AWS - rules controlling traffic to/from instances, act like host firewalls). 13 Regions: Geographic segmentation for performance, compliance, defense-in-depth. 14 Peer-to-Peer (P2P): Network architecture where nodes share workload equally. Used for file sharing. Hard to segment/control centrally. 15 Air Gap: Physically isolating systems from any network connection. Highest level of segmentation. Vulnerable to physical/insider threats (e.g., Stuxnet via USB). 16 Deperimeterization/Zero Trust: Security model assuming no implicit trust based on network location. “Never trust, always verify.” Requires strong authentication/authorization for every access attempt. Moves away from perimeter-based trust. 17 Architecture Contexts Requiring Segmentation/Specific Controls: * Cloud: Requires reliance on CSP controls, VPCs, Security Groups. Loss of traditional perimeter. * Remote Work: Extends perimeter. Need VPNs, endpoint security, user awareness, adapted policies. * Mobile: BYOT challenges. Need MDM, App Wrapping, Remote Wipe, VPN, Context-Aware Mgmt, secure config profiles. Address SCEP weakness, unsigned apps, sideloading risks. * Outsourcing/Contracting: Requires vendor assessment, SLAs, clear data ownership/sovereignty, segmentation of vendor access. * Wireless/RF Networks: Need secure protocols (WPA3), MAC filtering (optional), proper configuration, WIDS/WIPS. * Merging Networks (M&A): Integrate different security postures, policies, technologies. Requires careful planning/assessment. * Peering: Direct connection between networks. Needs defined security agreements. * Cloud to On-Premises: Requires secure connection methods (VPN, Direct Connect), careful policy integration. * Data Sensitivity Levels: Drives segmentation strategy (e.g., place restricted data in highly segmented zone). * Cross-Domain: Connecting systems of different security levels requires controlled interfaces/guards. * Federation: Linking identity systems requires trust establishment and secure protocols (SAML, OpenID). * Directory Services: Central identity source, needs securing (LDAP, AD). * SDN: Centralized control plane needs securing. Offers potential for automated segmentation/policy enforcement.

⠀Objective 1.2: Infrastructure Security Design (Resiliency, Scalability, Performance) 1 Scalability: Ability to handle growth. * Vertical: Increase resources of existing system (CPU, RAM). * Horizontal: Add more systems/instances. * Autoscaling: Automated adjustment of resources based on demand (cloud). 2 Resiliency: Ability to withstand/recover from failures. * High Availability (HA): Minimize downtime. Achieved via redundancy, failover. * Diversity/Heterogeneity: Using components from different vendors/technologies to avoid single point of failure or vulnerability. * Course of Action Orchestration: Automated response sequences for recovery. * Distributed Allocation: Spreading resources geographically or logically. * Redundancy: Duplicate components (power, network paths, servers). * Replication: Copying data/VMs to alternate locations. * Clustering: Multiple nodes work together providing HA and/or load balancing. 3 Automation: Using tools/scripts for efficiency, consistency. * SOAR: Automates incident response workflows. * Bootstrapping: Automated initial configuration/setup of systems. 4 Performance: Optimizing speed and efficiency. * Containerization: Lightweight virtualization, packages app + dependencies. Faster deployment, resource efficient. * Virtualization: Running multiple OS instances on single hardware. Resource consolidation. * CDN (Content Delivery Network): Distributed servers cache static content closer to users, reducing latency. * Caching: Storing frequently accessed data locally (browser, server, CDN) to speed up retrieval.

⠀Objective 1.3: Secure Application Integration 1 Secure Foundation: * Baselines/Templates: Standardized secure configurations for applications/environments. * Secure Design Patterns: Reusable solutions to common security problems in software design. * Secure Coding Standards: Guidelines (e.g., OWASP) for writing code that avoids common vulnerabilities. 2 Vetting & Assurance: * Application Vetting Processes: Procedures for evaluating the security of applications (internal/third-party). * Validating Third-Party Libraries (SCA): Assessing risks in external code components (vulnerabilities, licensing). * Software Assurance: Activities ensuring software is secure and functions as intended. * Code Signing: Digital signature verifying publisher identity and code integrity. 3 Development Environment & Pipeline: * Sandboxing/Development Environment: Isolated space for coding and testing. * Defined DevOps Pipeline: Streamlined process integrating development and operations. * SecDevOps: Integrating security practices throughout the DevOps lifecycle. * CI/CD Pipelines: Automating build, integration, testing, delivery/deployment. * Versioning: Tracking changes to code/configurations. 4 Testing Methods: * SAST (Static): Analyze source code without execution. Finds implementation flaws (e.g., buffer overflows). * DAST (Dynamic): Test running application. Finds runtime issues (e.g., SQLi, XSS). * IAST (Interactive): Combines SAST/DAST concepts, often using agents during runtime testing. * Regression Testing: Ensure fixes/changes don’t introduce new issues or break existing functionality. * Unit Testing: Test individual code components/modules. * Integration Testing: Test how different components work together. 5 Integration Technologies & Considerations: * APIs: Secure management (API Gateway) and implementation are critical. * Middleware: Software connecting disparate systems; needs securing. * Enterprise Applications: Integrating CRM, ERP, CMDB, CMS requires careful security planning. * Integration Enablers: Directory Services (identity), DNS (naming), SOA (component interaction via services), ESB (middleware facilitating SOA). 6 Secure SDLC Integration: * Formal Methods: Mathematically proving correctness (high assurance). * Security Touchpoints: Requirements gathering, design, implementation (coding), testing, fielding, upgrades, disposal. * Development Approaches: Agile, Waterfall, Spiral (choose/adapt with security in mind). 7 Best Practices: * OWASP: Follow guidelines (e.g., Top 10). * Proper HTTP Headers: Implement security headers (HSTS, CSP, X-Frame-Options, etc.).

⠀Objective 1.4: Data Security Techniques 1 Data Loss Prevention (DLP) Controls: * Blocking: External media (USB), printing, RDP, clipboard access. * Restricted VDI: Limit data transfer capabilities in virtual desktops. * Data Classification Blocking: Prevent transfer of data based on its sensitivity label. 2 Data Loss Detection: * Watermarking: Embedding hidden identifiers in data to track leakage. * DRM (Digital Rights Management): Technology to control access/use of digital content. * Network Traffic Decryption/DPI: Inspecting encrypted traffic content (requires MitM capability, e.g., via proxy). * Network Traffic Analysis: Monitoring data flows for anomalies suggesting exfiltration. 3 Data Identification & Handling: * Data Classification, Labeling, Tagging: Assigning sensitivity levels (Public, Internal, Confidential, Restricted; or NIST Low/Mod/High). Applying labels/tags. * Metadata/Attributes: Using file properties for classification/control. 4 Data Obfuscation: Making data less sensitive/identifiable. * Tokenization: Replacing sensitive data with non-sensitive equivalent (token). Original data stored securely elsewhere. * Scrubbing: Removing sensitive data elements completely. * Masking: Hiding parts of sensitive data (e.g., XXX-XX-1234). * Anonymization: Removing PII to prevent identification of individuals. 5 Encryption: Protecting data confidentiality (at rest, in transit). Apply based on sensitivity/risk. 6 Data Life Cycle Management: Apply security controls throughout: Create -> Use -> Share -> Store -> Archive -> Destroy. 7 Data Inventory & Mapping: Understanding what data exists, where it resides, and how it flows. Essential for effective DLP and classification. 8 Data Integrity Management: Ensuring data accuracy and preventing unauthorized modification. 9 Data Storage, Backup, and Recovery: Secure storage practices, regular backups, tested recovery procedures. RAID for disk redundancy/performance.

⠀Objective 1.5: Authentication and Authorization Controls 1 Credential Management: * Storage: Password Repository Apps, avoiding insecure end-user storage, On-prem vs. Cloud considerations, Hardware Key Manager/HSM for secure key storage. * PAM (Privileged Access Management): Controlling/monitoring accounts with elevated privileges (admin, root). * Password Policies: Enforce Complexity, Length, Character Classes, History, Max/Min Age. Audit compliance. Avoid Reversible Encryption for storing passwords. 2 Federation: Linking identity across different organizations/domains. * Transitive Trust: Potential risk (A trusts B, B trusts C => A trusts C). Limit or manage carefully. * Protocols: OpenID (identity verification built on OAuth), SAML (XML standard for exchanging authentication/authorization data), Shibboleth (open-source SAML implementation). 3 Access Control Models: * MAC (Mandatory): System enforces access based on security labels/clearances (e.g., military classifications). Users cannot change permissions. * DAC (Discretionary): Owner of resource determines who has access. Common in standard OS (e.g., file permissions). * RBAC (Role-Based): Access based on user’s assigned role(s) within organization. Simplifies management. * Rule-Based: Access based on predefined rules (e.g., firewall ACLs - time of day, location). * ABAC (Attribute-Based): Access based on attributes of user, resource, environment. Most flexible/granular, complex. 4 Authentication Protocols: * RADIUS: Centralized AAA (Authentication, Authorization, Accounting). UDP. Commonly used for network access (VPN, Wi-Fi). * TACACS(+): Cisco alternative to RADIUS. Separates AAA functions. TCP. Often preferred for device administration. * Diameter: Successor to RADIUS. More reliable (TCP/SCTP), better for mobile IP. * LDAP: Protocol for accessing/querying directory services (like Active Directory). * Kerberos: Ticket-based authentication protocol used in Windows domains (Active Directory). Provides SSO within realm. * OAuth: Authorization framework. Allows application to obtain limited access to user’s data on another service without sharing credentials. * 802.1X: Port-based NAC standard. Used for authenticating devices connecting to LAN/WLAN before granting access. Requires authenticator (switch/AP), supplicant (client), auth server (e.g., RADIUS). * EAP (Extensible Authentication Protocol): Framework used within 802.1X to perform the actual authentication. Supports multiple methods (EAP-TLS, PEAP, EAP-TTLS). 5 Multifactor Authentication (MFA): Using >1 type of authentication factor. * Factors: Something you know (password), have (token, phone), are (biometric). * 2FA: Specific type of MFA using exactly two factors. * 2-Step Verification: Often implies two steps of the same factor type (e.g., password + code via email/SMS), generally weaker than true MFA/2FA. * In-Band vs. Out-of-Band (OOB): OOB uses a separate communication channel (e.g., phone call, SMS) for second factor, generally more secure. * OTP (One-Time Password): Password valid for single use. * HOTP: HMAC-based, counter synchronizes. * TOTP: Time-based, time synchronizes. 6 Hardware Root of Trust: Secure hardware component (e.g., TPM) that forms the basis for platform integrity and secure operations. 7 Single Sign-On (SSO): Authenticate once to access multiple applications/services. Relies on trust relationship (e.g., Kerberos, SAML, OpenID Connect). * JWT (JSON Web Token): Compact, URL-safe standard for representing claims securely between parties, often used in SSO/federation. 8 Attestation and Identity Proofing: Processes to verify the identity of a user or device during enrollment or authentication. Ensures the entity is who/what it claims to be.

⠀Exam Essentials Summary:

  • Architecture Foundations: Understand the need for holistic enterprise security, hardening principles, limitations of perimeter defense, and balancing security with business needs. Recognize key services like Load Balancers, IDS/IPS, WAF, NAC, VPN, DNSSEC, Firewalls/UTM, Proxies, SIEM, FIM, DLP, AV and their placement/purpose.
  • Segmentation: Master different segmentation techniques (VLANs, Screened Subnets/DMZ, Microsegmentation, Air Gaps, Data Zones, VPCs) and their application in various contexts (Cloud, Mobile, Remote Work, M&A). Understand Zero Trust principles.
  • Infrastructure Design: Explain Scalability (Vertical/Horizontal, Autoscaling), Resiliency (HA, Redundancy, Diversity, Clustering), and Performance (Containers, Virtualization, CDN, Caching) concepts. Know role of Automation (SOAR, Bootstrapping).
  • Application Security: Understand secure SDLC integration, testing types (SAST/DAST/IAST), secure coding standards (OWASP), third-party library validation (SCA), API security, and integrating enterprise apps (CRM, ERP).
  • Data Security: Know DLP techniques (blocking, detection via watermarking/DRM/DPI), data classification/labeling, obfuscation methods (tokenization, masking, anonymization), encryption, and data lifecycle management.
  • Authentication/Authorization: Differentiate access control models (MAC, DAC, RBAC, Rule-based, ABAC). Recognize key protocols (RADIUS, TACACS+, LDAP, Kerberos, OAuth, SAML, OpenID, 802.1X/EAP). Understand MFA concepts (factors, OTP, OOB) and Federation principles. Know credential management and PAM importance.
