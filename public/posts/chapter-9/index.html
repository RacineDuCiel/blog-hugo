Core Objective 1.6: Given a set of requirements, implement secure cloud and virtualization solutions. A. Introduction: Shifting Boundaries & Cloud Concepts

  • Changing Network Boundaries: Cloud computing, telecommuting (WFH), and outsourcing make traditional network perimeters harder to define and secure. Control shifts.
  • Cloud Computing Definition: Consumption model for IT services (servers, apps, storage) over the Internet. End-users often unaware of underlying details. Focus on on-demand/elasticity and pay-per-use.
  • Core Challenge: Ensuring cloud provider controls meet or exceed in-house security requirements and policies. Addressing loss of direct control.

⠀B. Virtualization Strategies 1 Core Concept: Technology sharing physical computer resources (CPU, RAM, disk) among Virtual Machines (VMs). Improves resource utilization over dedicated physical servers. 2 Hypervisor (Virtual Machine Monitor - VMM): Software/firmware/hardware enabling virtualization. Creates and manages VMs, allocates resources. * Functions: Interfaces with hardware, intercepts system calls, operates with OS, provides hardware isolation, enables multi-environment protection. * Examples: VMware Workstation/vCenter, Oracle VM VirtualBox, Microsoft Hyper-V, XEN. 3 Hypervisor Types: * Type 1 (Bare Metal): Runs directly on host hardware. No underlying host OS needed. (e.g., VMware ESXi, Hyper-V role). More common in datacenters. * Type 2 (Hosted): Runs on top of a conventional host OS (e.g., Windows, Linux). Guest OS runs within hypervisor. (e.g., VMware Workstation, VirtualBox). Easier setup, often used for desktops/testing. 4 Host vs. Guest OS: * Host OS: OS of the physical machine running a Type 2 hypervisor. * Guest OS: OS running inside a VM. 5 VM Creation/Conversion: * Tools exist to convert physical machines to virtual images (P2V - e.g., VMware vCenter Converter) and virtual to physical (V2P). 6 Virtualization Sprawl: Too many VMs become unmanageable. * Mitigation: Strict policies, automation, VM library (with librarian role). 7 Other Virtualization Types: * Mainframe VM: Sharing resources on mainframes (e.g., IBM z/OS on AWS). * Parallel VM: Distributing tasks across multiple physical machines appearing as one environment. * OS Virtualization: Standard VM hosting guest OS (VMware, XEN, VirtualBox). 8 Containers: * Concept: Isolate applications (not entire OS) using OS kernel features (process isolation, resource management). Lower overhead than full VMs. * Benefits: Faster deployment, accelerated development. * Technology: Docker (popularized), Kubernetes (orchestration), Vendor platforms (Tanzu, AKS, ECS). 9 Emulation: * Concept: Mimicking hardware/architecture of one system on another (e.g., running console games on PC, Windows on Mac). * Characteristics: Often slower than the system being emulated. * Examples: Dolphin Emulator, Parallels. 10 Application Virtualization: * Concept: Allows access to applications not installed locally. Application is encapsulated from the underlying OS. * Benefits: Consistent experience across OS/browsers. * Example: Java Virtual Machine (JVM) allows Java apps to run anywhere JVM is installed. 11 VDI (Virtual Desktop Infrastructure): * Concept: Centralized desktop OS (e.g., Windows 11/Server) hosted on servers, delivered to endpoints via protocols (RDP, Citrix). Evolution of thin clients. * Benefits: Reduced onsite support, centralized management. * Disadvantages: Significant backend infrastructure investment. RDP can be an attack vector if not secured.

⠀C. Security Considerations for Virtualization 1 Security Advantages: * Server Consolidation: Fewer physical servers to protect. * Resource Optimization: Better hardware utilization. * Rapid Deployment/Testing: Easy creation/rollback for R&D. * Snapshots: Easy image backup/reversion points. * BCDR: VMs can be quickly moved/replicated to other hardware. * Reduced Space/Power: Fewer physical machines needed. 2 Security Disadvantages: * Complexity: Managing mixed OS environments, troubleshooting hypervisor issues. * Co-mingling of Data (Multi-tenant Risk): Data from different tenants/VMs on same physical host. Breach/seizure of one could impact others. * Physical Access Risk: Access to the physical host potentially grants access to all VMs. * Separation of Duties: Risk if network/security roles aren’t properly separated for virtual environment management. * Platform Misconfiguration: Errors in host/hypervisor config affect all VMs. * New Skill Requirements: Admins need virtualization expertise. * Host OS Vulnerabilities (Type 2): Underlying OS flaws can expose hypervisor and all VMs. * Data Migration Risks (Type 2): Shared folders between host/guest can allow malware transfer. * Privilege Escalation: Attackers gaining elevated access within host or guest OS. * VM Sprawl: Leads to unmanaged/unpatched systems. 3 Common Security Controls (Table 9.1): * Antivirus: On host and all VMs. * Hardening: Remove nonessential services/apps on host and VMs. * Physical Controls: Secure datacenter access. * Authentication: Strong access control for host/hypervisor/VMs. * Resource Access: Apply least privilege (limit admin accounts). * Encryption: For sensitive data at rest and in transit. * Remote Desktop Services (RDP): Restrict if unneeded; use strong encryption (256-bit+) if required. 4 IPsec VPNs: Can be used within VMs for secure communication. Components: AH (integrity/auth), ESP (confidentiality), SA (parameters/keys via ISAKMP/IKE).

⠀D. Provisioning, Deprovisioning, Middleware, Metadata 1 Provisioning: Preparing/equipping cloud service for new users. May involve startup fees. 2 Deprovisioning: Removing user/service access. May involve fees. 3 Middleware: Software layer between OS and applications providing communication/connectivity (e.g., between clients, servers, databases). Speeds deployment in cloud/containers. 4 Metadata & Tags: Information about cloud assets (data, instances). Used for organization, discovery, management, compliance, cost analysis. Tags are key-value pairs (e.g., data_owner=Finance, data_confidentiality=High). Crucial for governance.

⠀E. Cloud Deployment Models & Considerations 1 Cloud Deployment Models: * Public: Available to general public (e.g., Google Drive, AWS S3). CSP manages infrastructure. * Private: Operated solely for a single organization. Can be on-prem or hosted by 3rd party. More control, higher cost. * Hybrid: Mix of public and private clouds, linked by technology allowing data/app portability. Balances control and cost/scalability. * Community: Shared infrastructure for several organizations with common concerns (e.g., compliance, mission). Costs shared. 2 Hosting Models: * Multitenancy: Single software instance serves multiple customers (tenants). Resources shared. Lower cost, potential risk of data co-mingling/interference if isolation fails. * Single Tenancy: Dedicated software instance/infrastructure for one customer. Higher cost, better isolation/security, more customer management responsibility (updates). 3 Business Directives & Benefits: * Reduced Cost: Pay-as-you-go, less capital expenditure on infrastructure/maintenance. Scalability helps. * Increased Storage: Access to vast, elastic storage, often cheaper than on-prem. Regional redundancy often built-in. * Automation: CSP handles patching/updates for underlying infrastructure, reducing internal IT workload. * Flexibility: Access data/apps from anywhere. Supports mobility. * Data Protection: CSP responsibility for infrastructure security. Options for encryption, backups. * IT Focus Shift: Internal IT can focus on innovation instead of infrastructure management. 4 Key Considerations & Due Diligence: * Risk Assessment: Essential before moving services. Evaluate sensitivity, regulatory needs, loss of control tolerance, vendor viability. * “Try Before Buy”: Evaluate service functionality and performance. * SLA Review: Understand uptime guarantees, performance levels (CPU, memory, bandwidth), support response times, penalties. * Cost Analysis: Include provisioning/deprovisioning fees, usage costs. * Proprietary vs. Open Source: Impacts lock-in, portability. * Rollback Plan: How to revert if cloud migration fails. * BCDR: Understand CSP’s plan and how it integrates with yours. * Data Encryption & Isolation: Verify capabilities. * Forensics: Cloud complicates investigations (chain of custody, multi-tenancy). 5 Cloud Provider Limitations & Risks: * Downtime/Outages: Reliance on CSP availability (AWS 2021 outage example). * Privacy Concerns: CSP handling of data, potential access by CSP employees or via legal requests. * Limited Control: Less control over underlying infrastructure compared to on-prem. * Vendor Lock-in: Difficulty switching providers due to proprietary tech or migration costs. * IP Addressing: CSP may limit usable IP ranges; schemes might differ from on-prem. Often dynamic. * VPC Peering Limits: Restrictions on connections between VPCs (routing, only one connection allowed). * Service Termination: CSP might discontinue services (Samsung Cloud example). Requires user migration. * Terms of Service: Can change, may contain unexpected clauses (Apple/AWS examples). Read carefully. * Misconfiguration: Major risk source. Complex environment prone to user error (e.g., public S3 buckets). CSPs offer tools (Cloud Conformity Checks, CloudTrail) to help detect.

⠀F. Extending On-Premises Controls to the Cloud 1 Trust Shift: Must trust the CSP for infrastructure security. Verify their controls. 2 Key Security Questions for CSPs: * Regulatory Compliance: Can CSP meet requirements (ISO 27002, ITIL, COBIT, GDPR, HIPAA, PCI DSS, SOX, FERPA, etc.)? Willingness to undergo audits? * Data Access: Who at the CSP can access data (logical/physical)? Insider threat controls? * Data Classification: Does CSP use data classification? How is data separated? * Encryption: Type used (AES 128/256?) for data at rest/in transit? Key management practices? Virtual Private Storage option? * Employee Training: What security training do CSP employees receive? * SLA Terms: (Covered above). * Right to Audit: Can you audit the CSP’s controls/records? * Long-Term Viability: Financial health? Data return policy on failure? * Incident Response: CSP’s breach notification process? CSIRT capabilities? Law enforcement involvement policy? * BCDR: CSP’s plan? Data remanence handling for decommissioned hardware (destruction needed)? 3 Data Sovereignty: Data subject to laws of the country where it’s stored. Critical for compliance (GDPR example). May conflict with user’s location laws. Analyze legal issues (licenses, regulations, IP) and jurisdiction impacts. Encryption export controls may apply.

⠀G. Cloud Computing Vulnerabilities

  • Authentication Attacks: Exploiting weaknesses in login processes (XSS, CSRF).
  • Denial of Service (DoS/DDoS): Overwhelming services with traffic. Cloud scale can be a target. Botnets often used (LOIC tool).
  • Data Aggregation/Isolation Issues: CSP potentially reselling aggregated data? Insufficient isolation between tenants in multi-tenant environments.
  • Data Remanence: Risk of data recovery from improperly decommissioned media. Need secure sanitization/destruction (DBAN tool).
  • Data Exfiltration: Backups, backdoors, covert channels, social engineering targeting CSP employees, trojaned USBs, malicious insiders applying for jobs.
  • Misconfiguration: Leading cause of cloud breaches (user error in configuring services, permissions, storage access).

⠀H. Cloud Storage Models & Security 1 Enterprise Storage Needs: Handling massive data volumes (government, finance, e-commerce). Requires high availability, performance, security. 2 Storage Technologies: * Object/File-Based Storage (NAS): Appears as file server. Uses protocols like NFS, CIFS/SMB. (e.g., AWS S3). Hierarchical. * Block Storage (SAN): Appears as local disk/volume. High performance, low latency. (e.g., AWS EBS). * Database Storage: Specific storage optimized for databases. Redundancy via shadowing, remote journaling, electronic vaulting. * Blob Storage (Binary Large Object): Storing large unstructured binary data objects. * Key-Value Pairs: Simple data store associating a unique key with a value. Highly scalable. (e.g., Amazon DynamoDB). * Virtual Storage/File Sharing: Cloud services for storing/sharing files (OneDrive, Google Drive, Dropbox). Risks: Assets outside perimeter, loss of control, vendor viability, illicit data transfer, user-managed security. 3 Storage Area Network (SAN): Network dedicated to storage traffic. Fabric = SAN structure. 4 Virtual SAN (VSAN): Logical segmentation within a physical SAN fabric. Improves utilization, management. 5 Redundancy/Resiliency Techniques: * RAID: Disk redundancy/performance (older standard). * Dynamic Disk Pools (DDPs): Shuffle data/parity/spare capacity across drives. Faster rebuilds than RAID. * Location Redundancy: Storing data in multiple geographic locations (offsite backups). * Replication: Copying data (often real-time) for fault tolerance. * Multipath Solutions: Multiple routes between server and storage for availability/performance. * SAN Snapshots: Point-in-time copies for quick backup/recovery. 6 Storage Efficiency: * Data De-duplication: Storing only one unique copy of redundant data, using pointers. Saves space. 7 Advanced Storage Security Configurations: * Data Dispersion: Distributing data chunks across multiple cloud storage locations (pods). * Bit Splitting: Encrypt data -> Split into chunks -> Distribute chunks to different locations -> Re-encrypt chunks at new locations. Enhances confidentiality. 8 General Storage Security Best Practices: * Apply data protection policies. * Limit access (least privilege), implement classification. * Prevent exfiltration (e.g., block copying to unmanaged devices). * Audit critical configurations. * Encrypt sensitive data. 9 Removable Media Risks: USB OTG (easily lost/stolen), swappable drives (theft, unauthorized data transfer).

⠀Core Objective 3.4: Explain how cloud technology adoption impacts organization security. A. Automation and Orchestration

  • Automation: Tasks performed without human intervention (based on predefined rules/conditions). Reduces errors, improves consistency.
  • Orchestration: Coordinating multiple automated tasks into a larger workflow.
  • Benefits: Improved SOPs, efficiency, consistency.
  • Tools: Puppet (provisioning, config mgmt, Linux/Win), Chef (compliance/security focus), Ansible (simple orchestration, easiest setup).

⠀B. Encryption Configuration

  • Importance: Essential for cloud data security (rest & transit) due to increased exposure. Addresses compliance needs (PCI, HIPAA).
  • Kerckhoffs’s Principle: Security relies on key secrecy, not algorithm secrecy.
  • Data in Transit: HTTPS (using TLS) is standard. Encodes traffic between user and service.
  • Data at Rest: More complex. Options depend on CSP and customer risk tolerance.
  • Key Ownership Models:
    • CSP Managed: CSP controls keys. Easiest, but highest risk (insiders, external attacks on CSP). (Only 1% support tenant-managed keys per text).
    • BYOK (Bring Your Own Key): Customer generates/manages keys, uploads to CSP. CSP can access keys. Balance of control/convenience.
    • HYOK (Hold Your Own Key): Customer generates/manages/stores keys outside the cloud. CSP cannot decrypt data. Most secure, most complex management.

⠀C. Logs

  • Importance: Critical for security monitoring, incident investigation, compliance, audit trails. Show who did what, when, and system response.
  • Sources: Routers, switches, firewalls, servers, workstations, cloud services (e.g., AWS CloudTrail for API activity).
  • Challenges: Volume (Big Data), Collection, Storage, Encryption, Parsing (Normalization), Correlation across diverse sources/time zones.
  • Time Synchronization: Essential for correlation. Use NTP (Google Public NTP example).
  • Key Considerations for Logging Strategy:
    • Normalization/Parsing: Structure logs for analysis.
    • Correlation: Connect events across systems to identify incidents.
    • Monitoring & Alerting: Proactive detection of specific events/anomalies.
    • Availability & Storage: Secure, searchable, sufficient retention, potentially compressed.
    • Analysis: Manual and automated techniques.
  • SIEM Tools: Centralize log collection, analysis, correlation, reporting, visualization (ELK stack, Splunk, Rapid7 InsightIDR, LogRhythm, Exabeam). Require baseline period. Use logging infrastructure (syslog, syslog-ng). Capture logs, flows, metadata.
  • Impact of No/Poor Logging: Attackers operate undetected (Verizon DBIR ref: espionage, remote access attacks).

⠀D. Monitoring Configurations

  • Scope: Monitor network connectivity, VM status/access, database queries/access/integrity, website uptime, storage performance/cost.
  • Best Practices: Define key metrics, choose right tools, centralize monitoring, automate tasks, monitor end-user experience, test for failures, track costs.
  • Continuous Compliance: Use tools (e.g., Puppet) to monitor config drift against policy/code, automate remediation.

⠀E. Key Life-Cycle Management

  • Need: Manage cryptographic keys from creation to retirement.
  • Frameworks: NIST SP 800-57 Part 2, Rev 1 provides guidance.
  • Microsoft Model States (Example): Creation -> Initialization -> Full Distribution -> Active -> Inactive -> Terminated.
  • Policy: Requires defined Key Management Policies and Key Usage Policies (what assets/apps use keys for what operations).

⠀F. Backup and Recovery Methods

  • Shift in Strategy: With IaaS/Containers, focus shifts from backing up full systems to backing up code (infrastructure as code) and key data. Traditional methods may not apply directly.
  • Review/Testing: Regularly test DR plans and backup procedures against new frameworks.
  • Offsite Storage: Essential for geographic diversity (physical tapes/drives or cloud-based). Distance matters (avoid single regional disaster).
  • BCDR Plan: Documented processes/procedures for disaster response. Must account for cloud assets. Includes team, policies, exercises, technical infrastructure, comms plan. Primary vs. Alternative CSP BCDR options.

⠀G. Infrastructure vs. Serverless Computing

  • IaaS: Customer manages OS, middleware, apps. More control, more responsibility.
  • Serverless: CSP manages all infrastructure. Customer provides code, pays per execution. No server management needed. Faster development, potentially lower operational overhead for specific workloads. Examples: Workflows, Kubernetes services, hosted app environments.

⠀H. Software-Defined Networking (SDN)

  • Concept: Decouples network control plane (decision-making) from data plane (forwarding hardware) using a central software controller. Virtualizes the network.
  • Benefits: Centralized management, dynamic configuration, traffic shaping, flexibility, automation, programmability, visibility. Treats network as logical entity.
  • Layers: Application, Control, Infrastructure/Data Plane.
  • OpenFlow: Standard interface between controller and infrastructure layers.
  • Use Case: IaaS environments.

⠀I. Misconfigurations

  • Definition: Incorrect system configuration leading to security risks.
  • Impact: Major cause of cloud breaches and financial damage (IBM cost report). Often due to complexity and lack of user understanding.
  • Examples: Publicly accessible storage (AWS S3), misconfigured APIs.
  • Detection/Prevention: Cloud provider tools (AWS Cloud Conformity Checks), configuration auditing tools (AWS CloudTrail), proper training for admins.

⠀J. Collaboration Tools & Associated Risks

  • General Concern: These tools form company backbone but introduce security risks.
  • Web Conferencing: Eavesdropping risk (sniffing tools: Ettercap, Cain & Abel, UCSniff, VideoSnarf).
  • Videoconferencing: Sensitive info leak risk, webcam spying (malware like gh0st Rat). Mitigation: User training (attachments, installs), AV, strong/unique meeting passcodes.
  • Audio Conferencing: Risk of ambient noise being broadcast.
  • Instant Messaging (IM - Teams, Slack, etc.): Malware vector (worms, Trojans), sensitive data leakage (file transfer), impersonation, DoS. Policy Needs: Define allowed tools, encryption requirements, usage scope (internal/external), logging/compliance, file transfer controls, content filtering, user training. SPIM (Spam over IM).
  • Desktop/Application Sharing:
    • Risks: Unauthorized remote control (malware like Back Orifice, NetBus, Poison Ivy), exposing sensitive desktop contents, credential capture (Pass the Hash technique during remote assistance).
    • Mitigation: Block default ports (see Table 9.2), restrict access, avoid public advertising/indexing, use warning banners, terminate sessions properly.
  • Presence (Apple): Remote file access/sharing between Apple devices.
  • Domain Bridging: Risk of device connected to corporate network (VPN/remote assistance) also being connected to another network, creating unauthorized bridge.
  • Email:
    • Vulnerabilities: Clear text transmission, spoofing, spam, phishing, malware delivery.
    • Protocols & Ports: SMTP (25, 587/465), POP3 (110, 995), IMAP (143, 993). Secure versions use SSL/TLS.
    • Security: Policy prohibiting sensitive data via clear text. Mandatory encryption (PGP, S/MIME). Gateway filtering. User training.
  • Telephony (Legacy PBX):
    • Risks: Default password exploits (toll fraud - call collect to hacked line), Slamming (unauthorized carrier switching), Cramming (unauthorized charges).
    • Mitigation: Change default passwords, enforce user password changes, manage voicemail for departing employees. FCC banned slamming/cramming (2018).
  • VoIP: Runs voice over data networks (convergence).
    • Risks: Packet-switched vulnerability (On-Path attacks, sniffing - Cain & Abel), DoS (UDP reliance), potential C&C mechanism for botnets, SPIT (Spam over Internet Telephony).
    • Implementation Issues: QoS (Jitter, Latency) affected by encryption overhead. Softphone vulnerabilities (host exploits). Hardware phone vulnerabilities (embedded computers). Open standards aid attackers.
    • Best Practices: Strong auth, access control, disable services/ports, encrypt traffic, deploy FW/IDS, patching. Use VLANs to segregate voice/data (mitigates sniffing - VOMIT tool).

⠀K. Data Dispersion & Bit Splitting

  • Data Dispersion: Storing data chunks across multiple geographically distributed cloud locations (pods) for redundancy and potentially performance/compliance.
  • Bit Splitting: Encrypting data, splitting it into chunks, distributing chunks to different locations, and re-encrypting them. Makes individual compromised chunks useless. Enhances confidentiality.

⠀Exam Essentials Summary:

  • Virtualization: Differentiate Type 1 (bare metal) vs. Type 2 (hosted) hypervisors. Understand Containers (app isolation), Emulation (mimic hardware), and Application Virtualization. Know security pros/cons of virtualization and key controls (AV, hardening, access control, encryption). Recognize VM sprawl risk.
  • Cloud Models: Distinguish Deployment models (Public, Private, Hybrid, Community), Hosting models (Multitenant vs. Single-tenant), and Service models (IaaS, PaaS, SaaS, SECaaS).
  • Cloud Risks & Considerations: Understand the impact of shifting boundaries, loss of control, vendor lock-in, multi-tenancy risks, importance of SLAs, Right to Audit, vendor viability, data sovereignty (GDPR), and the major risk of misconfiguration. Perform due diligence.
  • Extending Controls: Ensure cloud security meets/exceeds on-prem standards. Verify CSP controls for compliance, data access, classification, encryption, training, IR, BCDR.
  • Storage Security: Know storage types (Object/File, Block, DB, Key-Value) and associated cloud examples (S3, EBS, DynamoDB). Understand redundancy (RAID, DDP, Replication, Location), efficiency (De-duplication), and advanced techniques (Data Dispersion, Bit Splitting).
  • Cloud Security Operations: Understand the role of Automation/Orchestration (Puppet, Chef, Ansible), need for robust Logging/Monitoring/SIEM (ELK, Splunk), Key Management lifecycle and ownership models (CSP, BYOK, HYOK), and evolving Backup/BCDR strategies for cloud/virtualized environments.
  • SDN: Recognize the concept (decoupling control/data planes), benefits (centralized mgmt, flexibility), and OpenFlow standard.
  • Collaboration Tools: Be aware of the specific security risks associated with web/video/audio conferencing, IM, desktop/app sharing, email, legacy telephony, and VoIP (sniffing, DoS, C&C, SPIT, QoS). Know basic mitigation techniques (encryption, policy, user training, segmentation).
